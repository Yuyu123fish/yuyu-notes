# 基本信息
项目简介：悦享生活态是一个面向同城用户的 O2O 本地生活服务平台，提供商户探店图文分享、高并发优惠券秒杀、帮助商家推广优惠信息等核心功能。
技术栈：SpringBoot + MySQL + MyBatis-Plus + Redis + Caffeine + Redisson + RabbitMQ
- 秒杀防超卖：使用Redis存储库存和订单信息，用Lua判断用户下单资格，防止超卖同时保证一人一单；
- 秒杀流程优化：用户下单后使用 RabbitMQ消息队列 异步扣减库存和生成订单，削峰填谷，异步解耦 ，提高并发性能；
- 支付和关单功能：使用乐观锁解决订单支付和关单的并发问题；
- 缓存优化：使用逻辑过期解决Redis热点Key的缓存击穿问题；使用缓存空值解决Key的缓存穿透问题；
- 多级缓存：使用 Caffeine本地缓存 与 Redis缓存 搭建二级缓存，提高热点数据的访问速度，降低Redis压力；
- 数据一致性保证：采用 Cache-Aside策略 先更新数据库后删除缓存，若删除失败消息队列补偿重试，TTL兜底共同保证数据一致性；
- 滑动窗口限流：使用 Redis + AOP + 注解 实现限流，支持全局、IP、用户多维度，防止系统过载，刷券、爬虫；

# 秒杀防超卖
使用Redis存储库存和订单信息，用Lua判断用户下单资格，防止**超卖**同时保证**一人一单**；

## 背景
平台上有折扣力度很大的优惠券，在某个时间点**限量**供应，届时会有很大的**高并发**流量。
所以得防止超卖和保证每个用户只买一次。
即**库存防超卖**和**一人一单**。

## 问题与方案
要达到防超卖和一人一单的目的，我们就需要做好检验，在正式下单之前确定**库存充足**且用**户具有下单资格**，那么这个操作是必须保证原子性，也就是在一个状态下执行，而且针对多线程的情况必须加入分布式锁来防止并发问题。
针对以下两个问题：
- 库存防超卖：
	- 可以在更新数据库的时候使用`update where stock > 0`来检验，但是会加锁，对应高并发不友好。
	- 也可以在redis中存入对应的剩余数量，然后业务代码在执行的时候做校验。此方案性能好。但需要引入分布式锁，防止出现线程安全问题。
- 一人一单：
	- 去数据库查是否下过单，针对一个用户发起多次下单请求，加入分布式锁保证并发安全。
高并发场景下直接查询和更新数据库，再加上**分布式锁**，会对性能造成严重影响。

因此考虑进行优化，在**redis里面做防超卖和一人一单检验**：
- 当请求进入到后端时，先在redis里面做检验，先判断库存是否大于0，不大于就返回错误码，否则进入下一步，判断用户是否下过单，在redis里面使用set类型（key为业务id+优惠券id，value为用户id，对于多单的，可以用hash），检查是否存在即可。存在则返回另一个错误码，否则就进行库存扣减和下单。
- 对于库存扣减和下单，在判断用户具有下单资格后，扣减库存，同时将用户id加到set中。
- 下单之后发送消息到**消息队列异步生成订单**。
- 数据库可以使用`update`来对满足条件的订单做**兜底**。

所以就不涉及到mysql，减小了mysql的压力，整个过程在redis中完成，redis执行lua脚本是**单线程**的，所以可以保证**两个动作的原子性**，也不需要加分布式锁了，大大提高了业务的秒杀性能。

## 追问
**1. Redis 扣了库存，但异步下单失败怎么办？**  
答：需要补偿机制。比如消息队列要保证可靠投递和消费，失败可以重试；如果最终失败，需要回滚 Redis 库存和用户下单记录，或者通过定时任务扫描异常订单状态做补偿。
**2. 为什么不用数据库直接扣库存？**  
答：数据库可以做，但高并发秒杀下数据库压力大，锁竞争严重。Redis 更适合做前置流量削峰和资格校验，数据库做最终落库和兜底。
**3. Lua 为什么能保证原子性？**  
答：Redis 执行 Lua 脚本时会把整个脚本作为一个整体执行，中间不会被其他命令插入，所以多个 Redis 操作可以具备原子性。
**4. 一人一单只靠 Redis Set 够吗？**  
答：Redis 可以挡住绝大多数重复请求，但数据库最好加唯一索引，比如 `user_id + voucher_id`，防止 Redis 异常或重复消费导致重复订单。
**5. Redis 和数据库库存不一致怎么办？**  
答：秒杀前把库存预热到 Redis，秒杀过程中 Redis 作为前置库存。最终以数据库落库为准，同时可以通过消息可靠消费、失败重试、补偿任务来保证最终一致。

# 秒杀流程优化
用户下单后使用 RabbitMQ消息队列 异步扣减库存和生成订单，削峰填谷，异步解耦 ，提高并发性能。

## 背景
平台上会发布折扣力度很大的优惠券，是在某个时间点限量供应的，所以会有很大的流量流入，也就是在秒杀后，会扣减mysql库存并生成订单。

## 问题与方案
实际上扣减库存和生产订单是**两个mysql操作**，如果说串行执行，那么会对秒杀的性能带来影响，而且再判断用户具有下单资格后，我们**库存**信息和用户下单信息都**存在了redis里面**，所以就可以下单后发送一个RabbitMQ消息到消息队列，然后在**消费者的逻辑里面去扣减mysql库存和生成订单**（使用事务来保证同时成功或者同时失败，失败做补偿重试），这样就把检验下单资格和生成订单**解耦**了。提高秒杀场景的并发性能。而且也适合后期做业务的功能扩展。

**兜底策略**（redis和mysql数据一致性）：比如redis扣减库存了，但是mysql却没有扣减和生成订单。那么针对这种情况，可以使用定时任务去查redis和mysql中的订单数据，发现不一致的话就做消息队列补偿重试或者人工介入。

# 支付和关单功能
使用乐观锁解决订单支付和关单的并发问题

## 背景
我们平台上的优惠券在用户下单后，需要在一定时间内进行支付，但此时面对两种状态的流转情况，一种是用户到时间未关闭，另一种是用户到时间刚好支付。

## 问题与方案
这两种情况同时发生的话，会带来并发安全问题，比如支付了但显示支付失败，未支付却显示支付成功。

有以下两种方案，也是对乐观锁和悲观锁的取舍：
- 乐观锁：在执行支付和关单的sql语句的时候，都使用`update status = '成功/失败' where status = '未支付'`，那么也就是说，最终只会有一条sql语句的`where`执行成攻，而另一条会失败。那么如果是支付成功，则正常支付。如果是失败，则支付失败，并且使用消息队列异步执行对应的逻辑代码，如果发现用户已经支付了，还需要进行退款操作，同时更新对应的库存信息和下单信息。
- 悲观锁：先锁住当前的订单，然后进行判断是否支付，最后再更新订单状态。

当前场景下，由于支付和关单同时出现是小概率事件，乐观锁很适合当前情况。而且还有可能是批量扫数据库查出来的关单，所以说如果使用悲观锁加锁判断的话，对性能会有一定的影响。
所以采用了乐观锁的方案。

## 追问
**1. 如果支付成功了，但订单已经被关了怎么办？**  
答：本地更新订单状态时会失败，因为 `status != UNPAID`。如果支付平台已经扣款成功，就需要走退款流程，或者进入补偿任务处理。  
**2. 为什么不用悲观锁？**  
答：支付和关单刚好并发是小概率事件，乐观锁成本更低。悲观锁会增加数据库锁竞争，批量关单时性能更差。  
**3. 怎么判断谁成功了？**  
答：看 SQL 影响行数。影响行数为 1 说明状态更新成功，影响行数为 0 说明状态已经被其他流程修改。  
**4. 关单成功后要做什么？**  
答：释放库存、恢复 Redis 中的秒杀资格或订单标记，具体可以通过 MQ 或补偿任务异步处理。  
**5. 这个算不算乐观锁？**  
答：算乐观锁思想。不是一定要有 version 字段才叫乐观锁，也可以通过状态字段作为 CAS 条件，只有当前状态符合预期时才允许更新。

# 缓存优化
使用逻辑过期解决Redis热点Key的缓存击穿问题；使用缓存空值解决Key的缓存穿透问题。

## 背景
平台的一些活动信息或者热搜榜单是热点数据，在活动上线之前需要先进行缓存预热，将其存入redis中，减少mysql的压力。

## 问题与方案
那么针对这些热点数据，就存在以下两个问题：
- 缓存击穿：在某个时间点，一个被高并发访问的热点key过期了，然后大量请求无法名字缓存，全部打到数据库，给数据库带来巨大压力。
- 缓存穿透：一些用户恶意查询我们缓存中不存在的数据，由于缓存无法命中，那么请求全部打到数据库，给数据库带来巨大压力。

那么针对**缓存击穿**，解决办法有以下几种：
- 一是使用**互斥锁**，多个请求去查数据库，那么在请求查库之前先进行加锁判断，拿不到锁就阻塞等待或者休眠一小会后尝试获取锁，拿到锁的就去查数据库并完成缓存重建。
- 另一个是是使用**逻辑过期**，也就是说针对热点数据我们不设置ttl，而是在字段里面加入一条过期时间的字段，当一个线程拿到热点数据后先判断是否过期，如果说过期了就进入查库阶段。那么这个阶段需要先获取互斥锁，拿不到锁的线程就全部返回旧数据，拿到锁的线程就去查数据库，完成缓存的重建并返回。
那么针对这两种方案的取舍，我们的热点数据并不是要求强一致性的，而是只要求最终一致性，所以我们使用逻辑过期这种方案。然后互斥锁这种悲观锁的方案，对于强一致性的场景，比如金融交易是合适的。

针对**缓存穿透**，解决办法有以下几种：
- 一种是**布隆过滤器**，这种方案是在我们的查redis之前，先到布隆过滤器里面做一次检验，如果发现查询的数据是存在的，则进入后续的流程，如果发现不存在，则直接返回。如果存在才查缓存和数据库。
	- 布隆过滤器的实现原理其实是通过对我们实际存在的商户id做多次哈希运算，然后存储在一个很长的存0和1位图中，后续的查询id经过哈希码运算后，发现对应的位上不位1，则不存在。那么它的缺点是无法删除商户信息，因为全是0和1，实际上有一种改进的办法，就是不存0和1，而是存出现次数，这种就可以进行删除操作，不过实现也比较复杂。
- 另一种是**缓存空值**，我们发现redis里面不存在，查数据库后发现也不存在，则在redis存入对应的数据，value设置为空，并设置一个较短的ttl做兜底策略，防止其长时间占用缓存，然后返回空值。
那么我们的项目采用了缓存空值的方案来解决缓存穿透的问题，因为我们本身的商户数据不是特别多，缓存空值也比较好实现。

## 追问
**1. 逻辑过期的缺点是什么？**  
答：会短时间返回旧数据，所以不适合强一致性场景，比如支付、库存这类核心交易数据。
**2. 逻辑过期为什么还需要锁？**  
答：锁不是为了保护读取，而是为了保证只有一个线程去重建缓存，避免多个线程同时查库重建。
**3. 布隆过滤器为什么有误判？**  
答：不同数据经过哈希后可能映射到相同位置，所以不存在的数据也可能被判断为“可能存在”。
**4. 缓存空值有什么问题？**  
答：如果恶意构造大量随机不存在的 Key，Redis 里会产生很多空值缓存，所以 TTL 要短，并配合参数校验、限流，严重场景可以加布隆过滤器。
**5. 热点 Key 一定不设置 TTL 吗？**  
答：逻辑过期方案里通常不依赖 Redis 的物理 TTL，而是在 value 中维护逻辑过期时间。但工程上也可以设置一个较长 TTL 作为兜底，避免长期无用数据一直占用内存。

# 多级缓存
使用 Caffeine本地缓存 与 Redis缓存 搭建二级缓存，提高热点数据的访问速度，降低Redis压力。

## 背景
对于一些存在redis里面的热点key，例如优惠券秒杀的详情页，它本身是更新频率极低，访问频率极高的，也就是读多写少的。

## 问题与方案
那么由于redis单个实例的性能有上限，如果许多个并发请求同时查redis，会给redis带来巨大压力，redis甚至会变成性能瓶颈。
所以引入了`caffeine`本地缓存来配合`redis`搭建二级缓存，它占用**JVM的堆内存**，只用来装热点数据（Caffeine 缓存的是详情页里的**静态/低频变更信息**，比如优惠券标题、活动时间、规则、展示信息。 
**不缓存**实时库存、用户是否下单这类**强一致或高频变化数据**），那么后续对于我们热点key的查询，就可以先查`caffeine`，查到了就返回，查不到就去查redis，存在的话就重建`caffeine`并返回，不存在则继续查数据库。然后更新先更新`redis`缓存再更新`caffeine`缓存。

那么加入`caffeine`本地缓存后，就引入了新的问题，就是本地缓存可能和redis，mysql存在数据不一致的问题，那么解决办法是比如利用**广播消息**通知对应的节点删除缓存。其实有一种**兜底策略**是给本地缓存设置一个**较短的ttl**，这样就可以不用去管本地缓存的更新。

## 追问
**1.Redis单实例压力过大，为什么不搭建Redis集群**
答：考虑成本问题，部署Redis集群的成本远高于本地缓存，并且本地缓存的速度比Redis缓存更快，所以可以优先选择本地缓存方案。
**2. Caffeine 放在 JVM 堆里，会不会占很多内存？**  
答：会，所以要设置 `maximumSize`、TTL、淘汰策略，只缓存热点数据，不能无脑缓存所有数据。
**3. 多节点下 Caffeine 怎么保证一致性？**  
答：强一致比较难，一般通过 MQ / Redis PubSub 广播删除本地缓存，再配合较短 TTL 兜底，保证最终一致。
**4. 为什么不只用 Redis？**  
答：Redis 虽然快，但仍然有网络 IO，而且热点 Key 高并发下 Redis 也可能成为瓶颈。本地缓存可以把最热的一部分请求直接挡在服务进程内。
**5. 哪些数据不能放 Caffeine？**  
答：库存、余额、订单状态、用户是否已下单这类强一致或高频变化数据不适合放本地缓存。Caffeine 更适合活动详情、配置、字典、规则、榜单快照这类读多写少数据。
**6. 本地缓存更新用更新还是删除？**  
答：多节点场景下更推荐删除缓存，让后续请求重新加载。直接更新很难保证所有节点同时更新成功。

# 缓存一致性保证
采用 Cache-Aside策略 先更新数据库后删除缓存，若删除失败消息队列补偿重试，TTL兜底共同保证数据一致性。

## 背景
我们商户数据是热点数据，所以放到了redis里面当作缓存。

## 问题与方案
由于mysql里面的数据是存在变动的，所以必须得保证mysql和redis的数据一致性。

那么有以下几种方案：
- 更新数据后**再删除缓存**（**cache-aside**），如果删除失败则发送一个消息到RabbitMQ消息队列做补偿重试，带上要删除的key的信息，如果消费者删除失败则进行**梯度重试**删除，**尽可能地保证数据地最终一致性**。
- 延迟双删，更新数据库前删除一次缓存，更新数据后延迟一个较短的时间再删除缓存。
- 使用Canal伪装mysql的从节点订阅binlog日志后，使用消息队列异步删除缓存。
最后我们选择了更新数据后同步删除的缓存的方式来保证缓存一致性，因为它实现简单，如果引入canal又多了需要维护的对象，它适合在大型项目里面使用。
当然，以上方案保证的都是数据的最终一致性，不能保证数据的强一致性，最后无论哪种策略，我们都需要**设置一个ttl来做兜底**。

## 追问
**1. 为什么是删除缓存，而不是更新缓存？**
因为更新缓存容易出现并发写覆盖问题，而且有些缓存数据可能是多个表聚合出来的，更新逻辑比较复杂。删除缓存更简单，下一次查询再从数据库加载最新数据，能减少双写不一致的风险。
**2. 为什么先更新数据库，再删除缓存？**
因为数据库是最终的数据源，先保证数据库更新成功，再让缓存失效。这样后续读请求最多短时间读到旧缓存，但缓存被删除后会重新加载新数据。如果先删缓存再更新数据库，可能在数据库还没更新完成时，有读请求把旧数据重新写回缓存。
**3. 先更新数据库后删除缓存还有没有并发问题？**
理论上还有极小概率问题。比如缓存刚好失效，一个读请求查数据库读到旧值，还没来得及写入缓存；这时写请求更新数据库并删除缓存；随后读请求把旧值写回缓存。这个概率比较低，可以通过较短 TTL、延迟双删、MQ 补偿等方式降低影响。我们项目里主要用 TTL 和删除失败补偿来保证最终一致性。
**4. MQ 补偿失败怎么办？**
可以让消费者重试，比如梯度重试；如果多次失败，可以进入死信队列或者记录失败 key，由定时任务继续扫描补偿。最终还有 TTL 做兜底，避免脏数据永久存在。
**5. Canal 方案有什么优缺点？**
优点是对业务代码侵入小，直接监听 MySQL binlog，数据库变更后异步删除对应缓存，适合大型系统或者多个服务共同修改数据的场景。缺点是引入 Canal、MQ 等额外组件，链路更长，运维复杂度更高。对于我这个项目来说，用 Cache-Aside 更轻量。

# 滑动窗口限流
使用 Redis + AOP + 注解 实现限流，支持全局、IP、用户多维度，防止系统过载，刷券、爬虫。

## 背景
有几种特殊情况，比如
- 一，在优惠券秒杀流量过大的时候，需要限流来保证系统的稳定性。
- 二，在领取无限制的优惠券时，为了避免恶意刷券，需要对用户来进行限流处理。
- 三，针对现在的爬虫爬取商户信息，需要对IP进行限流。

## 问题与方案
这个限流组件需要无侵入，高性能，容易使用这几个特点，那么有以下几种方案可以选择：
- **固定窗口**，这个在两个时间段的临界点会有流量突刺的情况，不考虑。
- **漏桶算法**，这个是以恒定的速率处理任务，做到削峰的效果，但是问题是漏桶如果满了，多余的任务会被丢弃，那么针对我们的秒杀，肯定不能说去拒绝用户的秒杀请求。
- **令牌桶算法**，以固定的速率生成令牌放到桶中，然后需要获取令牌才能继续处理。这种可以应对一定程度的流量突发，比如在用户和ai对话的时候做限流处理。但是坏处是如果有用户而已发起脉冲攻击，会导致令牌一下子全部消失，其他用户没有令牌可用的情况。
最终选择了**滑动窗口算法**，将时间分成多个窗口，统计滑动窗口里面的请求总数，随着时间推移窗口向前滑动，然后旧的窗口的数据会过期。那么它的有点是：
- 可用提供精确的控制。
- 还可以有效应对脉冲攻击，因为请求者在短时间内发起的请求只对一小部分窗口有效。

那么具体的流程细节是：
1. **自定义限流注解**：包含参数有：**限流key前缀，时间窗口大小，窗口内允许的请求数量，限流提示信息，限流维度（全局限流，用户限流，IP限流）**。
2. 实现**AOP切面**：获取注解上的参数（**限流key前缀，时间窗口大小，窗口内允许的请求数量，限流提示信息，限流维度**），然后传递参数给`lua`脚本并执行。
3. **限流`lua`脚本**：实现滑动窗口限流算法，使用Redis的ZSet（带有score分数）实现，内部流程如下：
	1. **删除超出时间窗口的数据**，执行`ZREMRANGEBYSCORE`，从有序集合中移除超出指定分数区间的成员。
	2. **获取当前时间窗口内的成员个数**，执行`ZCARD`返回有序集合中的成员个数，也是当前限流时间窗口内还存在的成员。
	3. 当 **当前窗口内的请求数量 小于 时间窗口内限制的次数** 时，则添加当前请求到ZSET内（用时间戳+随机数作为member，用时间戳作为score）并设置一个**比时间窗口略长的ttl**作为兜底。否则返回0表示限流。 
所以我通过滑动窗口限流的方式，针对高并发访问的方法执行全局限流，对于可用无限制获取的券主席用户限流，对于可能被爬虫的商户信息做了IP级限流。

## 追问
**1.为什么用 Lua？**
因为滑动窗口限流不是单条 Redis 命令能完成的，需要删除旧数据、统计当前数量、添加新请求。如果这些操作分开执行，并发情况下可能出现多个请求同时判断没超过限制，然后都放行的问题。Lua 可以把这些操作放到 Redis 端一次性执行，保证原子性。
**2.为什么用 ZSet？**
因为 ZSet 可以按照 score 排序，而我把请求时间戳作为 score。这样就可以很方便地删除窗口时间外的数据，并统计当前窗口内还剩多少请求。它适合实现比较精确的滑动窗口限流。
**3.这个方案有什么缺点？**
缺点是每个请求都可能往 ZSet 里写一条记录，在高 QPS 场景下 Redis 压力会比较大，尤其是全局秒杀限流。如果是特别高并发的核心链路，可以考虑令牌桶、漏桶、网关限流，或者本地限流加 Redis 分布式限流结合。我的方案更适合用户级、IP 级、接口级这种需要精确控制的限流场景。
**4.IP 限流有什么问题？**
IP 限流可能会误伤，比如同一个学校、公司、NAT 网关下很多用户共用一个公网 IP。另外如果服务前面有网关或者 Nginx，直接取 request remote address 可能拿到的是代理 IP，所以需要从可信网关传递的 header 中获取真实 IP，同时也要防止伪造。
**5.用户限流和 IP 限流怎么选？**
如果用户已经登录，优先用用户 ID 限流，因为更准确；如果是未登录接口，拿不到用户身份，就可以用 IP 限流。对于风险比较高的接口，也可以组合使用，比如同一个用户一分钟最多多少次，同一个 IP 一分钟最多多少次。
**6.秒杀场景限流了用户怎么办？**
限流后直接快速失败，返回“当前请求人数过多，请稍后再试”之类的提示。限流的目标不是保证每个请求都被处理，而是在流量超过系统承载能力时保护系统，避免数据库、Redis 或 MQ 被打垮。
