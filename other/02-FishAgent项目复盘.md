FishAgent 智能体
项目简介：基于 Spring Boot + Spring AI Alibaba 开发的全栈 AI 智能助手系统，集成自主规划智能体、三层分层记忆、多索引并发 RAG 及知识库 OCR 闭环。
后端技术栈：Java21 + SpringBoot + Spring AI Alibaba + MyBatis-Plus + Redis + Elasticsearch + MinIO(RustFS) + MySQL + Python
- 自主规划智能体：基于 ReactAgent 构建「思考-行动-观察」多步骤自主推理循环；针对大模型易陷入死循环的痛点，引入图节点切换硬上限、ModelCallLimitHook 优雅终止及原子状态机三重防死循环机制，保障系统高可用 ；
- 三层记忆分层设计：设计三层记忆架构，短期 Redis 滑动窗口同步加载，长期 ES 向量事实异步注入，完整对话记录落盘 RustFS/MinIO 。抽取与压缩链路职责严格交叉隔离，避免摘要重复入库污染向量索引；
- 虚拟线程并发 RAG： 私有/公共/知识库三索引物理隔离 。利用 JDK21虚拟线程执行 6 路并发检索召回，在异步线程中显式回放 UserContextHolder 快照，解决 ThreadLocal 丢失导致的私有数据空集问题，结果去重注入上下文 ；
- 知识库全链路闭环： 支持小文件直传与大文件分片上传，采用 Java落盘分发 + Python Worker异步消费 的解耦架构，执行“视觉 OCR 提取 → 文本分块 → 向量化 → ES 检索库”的解析流水线，并使用 Redis Stream 结合超时补偿定时任务彻底杜绝毒消息死循环；
- Tool Calling 扩展机制： 基于 SPI 机制自主设计 AgentToolProvider 工具注册中心 + ToolRegistry 自动发现，通过 @ConditionalOnProperty 按配置懒装配；新增工具仅需建类加 @Component，实现零侵入动态扩展；

- **日志分散**：给实例加上实例标识，打印在日志里面。

# 自主规划智能体
基于 ReactAgent 构建「思考-行动-观察」多步骤自主推理循环；针对大模型易陷入死循环的痛点，引入图节点切换硬上限、ModelCallLimitHook 优雅终止及原子状态机三重防死循环机制，保障系统高可用。

## 背景
单纯的大模型对话是只能一次对话一次输出，只能解决单个对话的问题。

## 问题与方案
因此为了让智能体去解决一些复杂问题，或者说是一些连续的多步骤问题，那么就需要让智能体具有思考-行动的能力，智能体每次思考后都会去决定下一步做什么，是使用工具还是说直接生成答案，如果选择使用工具，那么这一轮工具调用的结果会注入到下一次思考的提示词中，于是就让智能体可以多步骤的去解决一些问题。那么有ReAct模式，还有plan-execute模式，后者更适合做长期执行的任务，实际上常常将两者结合着来做。

那么我在项目中是使用了SpringAlibaba的graph**图节点状态机**来让大模型进行状态的流转，相比较传统的while(true)循环，图节点的可控性更高，因为我们不需要在while循环里面去编写大模型调用工具的逻辑，重试策略和异常处理流程。那么底层会将一个状态到另一个状态的转移，视为一个图节点到另一个图节点的移动，**可扩展性更高**。

那么这就引入了另外的问题，我们知道智能体在特殊情况下可能会触发一直思考的而无法终止的情况，比如一次工具调用异常后，再次调用此工具，由此形成死循环导致不必要的token浪费，当然我们也可以在提示词里面提示工具调用失败后禁止再次调用。我们的项目引入的方案如下：
- 图节点切换硬上限：因为每次状态切换都是在状态节点之间移动，所以我们为每个节点的进入次数做了上限处理，设置为最大大模型调用次数的4倍，达到最大值时会直接抛出异常兜底。
- 大模型调用次数上限（10次）：这是第二道防线，当大模型调用达到这个次数的时候会强行生成答案，做到优雅终止。
- 第三道防线是我们的业务层：我们在业务层里面留了接口，可以直接终止大模型的思考来直接生成答案，目前的应用是用户直接终止思考来让大模型生成答案。
那么基于这三重防死循环机制，很有效的避免了智能体无限死循环的机制。

## 追问
**1.recursionLimit 抛异常后，用户看到的是什么？你怎么处理的？**
理论上这道防线作为兜底防线是不会触发的，因为大模型调用十次后就达到上限了，所以更可能是原因是框架bug之类的。那么用户会看到error信息，代价就是此轮对话不会持久化，用户要重试。

# Tool Calling 扩展机制
基于 SPI 机制自主设计 AgentToolProvider 工具注册中心 + ToolRegistry 自动发现，通过 @ConditionalOnProperty 按配置懒装配；新增工具仅需建类加 @Component，实现零侵入动态扩展。

## SPI机制
SPI机制是一种**通过实现统一的接口定义来扩展相应功能**的解耦机制，目的就是新增核心功能的时候不用修改已有代码。

## 问题与方案
我们定义了一个基本的模板接口，也就是`ToolProvider`，它里面定义了name，build这两个方法以及一个是否启动的enable()，我们实际的工具通过实现此接口：
- 在Spring容器启动时，会通过`@ConditionalOnProperty`发现如果有工具出现例如api未配置等等问题，那么会跳过将它注册为Bean。
- 然后使用`@Component`注解将其注入到容器，然后会被我们的`ToolRegerstry`扫描到并注入，就会跳过那个工具，这样不同工具间就不会相互影响。
所以我们使用了SPI机制解耦了工具定义和注册这两个过程，让不同的工具之间不会因为异常而互相影响，同时提高了工具的扩展性。

## 追问
**1.为什么不在每次请求时动态构建工具，而是 @PostConstruct 一次性构建？**
工具是无状态的（纯函数式），构造一次复用即可，避免请求路径上的重复初始化开销
# 三层记忆分层设计
设计三层记忆架构，短期 Redis 滑动窗口同步加载，长期 ES 向量事实异步注入，完整对话记录落盘 RustFS/MinIO 。抽取与压缩链路职责严格交叉隔离，避免摘要重复入库污染向量索引。

## 问题与方案
由于每次发给大模型的都是当前的对话上下文窗口，所以我们需要让大模型记住我们当前对话之前的信息，以及部分需要长期记忆的信息，还有我们的知识库等等。

那么针对我们的**短期记忆**，我们采用了**滑动窗口+主动摘要**的方式，具体原因如下，如果使用直接截断的方式，那么肯定会丢失大量的上下文，这对用户的体验是很不友好的，还有一种办法是记录状态节点的信息，但它的 缺点也同样是可能会丢失我们一开始的背景信息。所以我们使用了滑动窗口+主动摘要的方式来解决这个问题。
那么流程如下，在大模型返回给用户对应的文本后，我们会进行一次是否需要**压缩摘要的检测**，如果没有必要就不进行，如果有必要压缩的话，就会发给当前的对话模型，提示词里面严格约束其做摘要，这个过程是**异步的**，用户几乎没有感知，总结摘要后替换原来的上下文。
这样就让大模型的对话可以更长和更久。

那么针对我们的**长期记忆**，我们是在对话完成后，额外使用一个**温度为0的模型**先判断是否需要进行记忆，同时针对一些不属于背景信息的信息，例如“你是xx，xx是什么产品”这种信息，也做了**严格的提示词约束**，这个过程也是异步的，模型判断后会将需要记入到长期记忆的文本使用向量模型向量化然后存入到我们的es中，这样就完成了用户的长期记忆，可以让大模型生成的内容更具有针对性。

对于对话历史的落盘，在每轮对话结束后，会写入到每个对话的json文件，写入时用`ReentrantLock`按`sessionId`增加锁粒度，并发安全。

## 追问
**1.为什么压缩和录入要用两个不同的 ChatModel Bean（memoryChatModel）而不是复用对话模型？**
对话模型需要高 temperature 保证创意性，压缩/录入需要低 temperature 保证准确性
压缩模型禁用 tool calling，避免压缩过程中触发工具调用
**2.完整对话存在 RustFS 里，如果 RustFS 挂了怎么办？**
RustFsChatMemoryStore 的写操作会抛异常（IllegalStateException），但 ChatService.persist() 里 catch 了异常不阻塞SSE done 事件返回。代价是这轮对话丢失持久化，但主流程（对话本身）不受影响。

# 虚拟线程并发 RAG
私有/公共/知识库三索引物理隔离 。利用 JDK21虚拟线程执行 6 路并发检索召回，在异步线程中显式回放 UserContextHolder 快照，解决 ThreadLocal 丢失导致的私有数据空集问题，结果去重注入上下文。

## 问题与方案
为了让用户的回答更具有针对性，所以使用了RAG检索增强生成获取到部分信息注入到背景提示词中，然后让大模型的回答更严谨。
那么这个决策过程是这样的，一开始我想的是引入纯向量做检索，那么它确实可以增强生成的结果的效果，但是对于一些可能字差别不大但语义却相隔很远的信息，可能会涉及到无法召回的情况，那么后来使用es的原因是，**es支持向量索引，同时其本身也支持文本做倒排索引**，所以就选择了ES做为我们的向量数据库兼文本库，那么就支持了**文本+向量**的双路召回。

（查询重写其实也有，但是没写）
其实这里也引入了**多查询扩展**，整句作为第一条子查询，再**按词边界切分提取关键词片段**，做多查询扩展，包装成查询有好的子句，最多12条，下面再进行6路并发检索。
那么我们的ES里面其实是有**三个索引**的，一个是用户的私有记忆，这是在用户的每轮对话结束后使用温度为0且没有工具的模型抽取的。一个是用户自己的上传的知识库，还有一个是管理员自己上传的知识库。所以这里就需要**对三个索引做文本+向量检索，那么也就是六路检索**。这里是**每个子查询做六路并发检索**（文本做了扩展查询，向量只有一路，也就是N个子查询对应N×3个文本+3个向量）。

然后本身我们的查库过程是一个**IO密集型过程**，所以可以使用JAVA21提供的虚拟线程来完成，虚拟线程是用户态的轻量级线程，由jvm管理，一个实际的线程可以挂载非常多个轻量级线程，非常适合做IO密集型任务，所以这里也使用了虚拟线程。

那么再后来遇到的一个**比较严重的问题**是，我们的用户id信息是存在`ThreadLocal`也就是主线程里面的，他和虚拟线程是数据隔离的，也就是虚拟线程是拿不到我们主线程里面的用户信息的，所以这里就需要对每个虚拟线程做了一次`ThreadLocal`的快照回放，那么就解决了用户信息丢失的问题。

然后就是进行**合并后去重**，再按**照分数排序**，截取**top8**的数据再注入到系统提示词中。

## 追问
1. 为什么不用 LLM 做多查询扩展（比如生成多个检索问题），而是用 BreakIterator 分词？
	- 零额外 LLM 调用成本，延迟更低
	- BreakIterator 支持 Locale.CHINESE，中文分词效果够用
	- LLM 扩展适合更复杂的场景，但当前规模下性价比不高
2. 12 个并发 ES 查询，性能扛得住吗？有没有做限流？
	- 虚拟线程池有固定大小（ragRecallExecutor），天然限流
	- 每个子查询的 perSubquerySize=5，单次返回量可控
	- 任何一路**失败都 catch 返回空列表**，不影响其他路
**3.查询重写的 ChatModel 和对话的 ChatModel 是同一个吗？**
是的，用的是容器内 @Primary 的 ChatModel，但温度调到了 0.1，并且通过严格 JSON 指令约束不让它"抢答"。
**4.为什么文本路按子查询检索，向量路只用完整句？**
文本检索（match）依赖关键词命中，拆开可以独立匹配不同片段
向量检索是语义级别的，整句 embedding 已经编码了完整语义，拆开反而丢失上下文
**5.如果某一路 ES 查询超时或异常，会怎样？**
safeTextSearch / safeVectorSearch 里 catch 异常返回空列表，不影响其他路
最终合并时空列表被忽略，只是少了这一路的召回结果

# 知识库全链路闭环（重点）
支持小文件直传与大文件分片上传，采用 Java落盘分发 + Python Worker异步消费 的解耦架构，执行“视觉 OCR 提取 → 文本分块 → 向量化 → ES 检索库”的解析流水线，并使用 Redis Stream 结合超时补偿定时任务彻底杜绝毒消息死循环。

## 问题与方案
一开始是准备直接在Java里面拿Tika做文档解析，但出现了一些问题，比如java语言很重，以及很多地方**处理非常麻烦**，同时由于解析过程中，对于文本会占用大量的堆内存，对jvm是不友好的，所以就开始思考有没有更轻量的办法，可以满足以下几个条件：
1. 语言轻量，方便解析。
2. 不占用jvm资源。
最后了解到可以**使用python来做异步解析**，那么就引入了另外一个问题，java和python之间如何通信，最后思考的项目本身用redis来存储我们的比如对话记忆，或者说sse对话锁之类的信息，而且redis本身支持stream消息队列，支持持久化，有消息确认机制这种可靠性保障，同时也无需额外引入组件来增加维护量，于是就考虑使用消息队列来联通我们的java和python这两个模块。

**那么java和python之间是完全没有侵入的**，它们完全共享底层组件，例如mysql，redis，minio这些，也就是说，我只启动java，那么可以完成文件的上传功能，发送消息到redis stream里面，然后落盘到我们的minio，同时写一条记录到我们的MySQL。那么只启动python的话，它里面的消费者（默认两个）会不断读取消息队列的消息，顺序如下：自己还没处理完的，别人超时120s没有ack的，新消息。然后就开始进行解析。

然后这里又遇到了问题，就是最开始我们是基于文本做解析的，从**一开始的pypdf到pdfminer，它们解析出来的文本都有非常大的问题**，要么是由于pdf字体不规范，解析出字符编码，要么是直接解析出来了乱码，所以后面考虑使用视觉ocr做文本提取。
然后**PyMuPDF将pdf转为图片**，这个过程虽然比文本慢了很多，但是精准度提高，而且我们的知识库上传功能本身也是异步的，用户并不会说有非常紧急的知识库使用需求。

那么视觉ocr基于300dpi提取出文本后，就开始做分块处理，我们使用的是512token（前后重叠50个token）进行分块，然后使用向量模型批量（25）转换，然后入库。最后完成对mysql中记录的修改。

这里有兜底策略，其实就是以下三种：
- worker临时崩溃，那么会重新认领并重新处理
- worker半截崩溃，写了一半没写，那么会清洗旧的切片再处理
- worker永久不可用，那么会java这边会定时进行扫描，扫描超时一定阈值（10分钟）的未处理完的数据做清理，同时mysql中记录的状态改为失败。
- **最后无论成功失败还是非法消息全都ack避免反复消费死循环**。

关于小文件直传：
前端判断文件大小后，直接走一个请求：
  POST /api/knowledge/upload  (multipart/form-data, file=xxx)
  后端 ingest() 方法一口气完成三件事：
  1. RustFS.putDocObject(minioPath, stream, size, contentType)
     → 文件直接写入对象存储，路径如 user/{userId}/{taskId}_{filename}

  2. MySQL INSERT (status=PENDING)
     → 写一条 document_metadata 记录
     → 如果插入失败，回滚删除 RustFS 上的文件

  3. Redis XADD fish:doc:ingest {task_id, minio_path, scope_type, user_id, ...}
     → 投递消息到 Stream
     → 如果投递失败，回滚：删 RustFS 文件 + MySQL 状态改 FAILED
  返回 { taskId }
  三步是顺序执行的，任何一步失败都会回滚前面的操作。
  
关于大文件分片：
前端分三阶段走：
  阶段一：初始化
  POST /api/knowledge/upload/init
  Body: { fileName, fileSize, contentType }
  后端只做两件事：
  - **生成 taskId + minioPath**
  - **MySQL INSERT (status=PENDING)**
  不写文件、不发 Stream。返回 { taskId, uploadId, minioPath } 给前端。
  阶段二：**逐片上传（循环）**
  POST /api/knowledge/upload/chunk  (multipart/form-data)
  Params: taskId, uploadId, minioPath, partNumber
  Body: chunk=xxx
  每个分片写入 RustFS 的 **staging 临时路径**：
  staging/{taskId}/1
  staging/{taskId}/2
  staging/{taskId}/3
  每片返回 { etag }，前端记住每片的 partNumber 和 etag。
  阶段三：合并完成
  POST /api/knowledge/upload/complete
  Body: { taskId, uploadId, minioPath, parts: [{partNumber, etag}, ...] }
  后端执行：
  1. RustFS.composeDocObject([staging/{taskId}/1, staging/{taskId}/2, ...], minioPath)
     → 把所有**分片合并**成最终文件
  2. **清理 staging 目录**
  3. **Redis XADD** fish:doc:ingest {task_id, minio_path, ...}
     → **投递 Stream**，后续流程和直传一样
  4. 返回 { taskId }
  如果合并失败 → 标记 FAILED + 清理 staging。如果 Stream 投递失败 → 标记 FAILED + 删除最终文件。
  还有一个取消接口 POST /api/knowledge/upload/abort，前端可以主动取消上传 → 清理 staging + 标记 FAILED。

## 追问
1. 为什么选 Redis Stream 不选 RabbitMQ/Kafka？
    - 项目已经依赖 Redis，不需要额外引入中间件增加维护成本
    - Stream 支持消费者组 + 持久化 + ACK 机制，满足当前规模
    - 缺点是 Redis Stream 不如 Kafka 适合海量消息，但知识库上传量级不需要
2. OCR 每页 1-3 秒，100 页的 PDF 要多久？怎么优化？
    - 100 页大约 100-300 秒。可以多 Worker 水平扩展（消费者组自动负载均衡）
    - 也可以把按页解析改成按页并发（Python 多进程），但当前是按文档粒度消费
  3. delete_by_doc_id 幂等清理是怎么保证的？
    - ES 的 delete_by_query 按 doc_id term 匹配删除，无论有没有数据都能执行
    - 重处理时先删旧的再写新的，保证不会出现重复切片
  4. Java 侧的上传流程是怎样的？直传和分片的分界线？
    - ≤ 1MB 直传，> 1MB 分片（每片 5MB）
    - 分片上传：前端先调 /init，再逐片 /chunk，最后 /complete 合并
