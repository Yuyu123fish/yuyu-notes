# 四层模型
- 应用层：**最上层**，各种协议都在这里，比如http，ftp，smtp，dns。
- 传输层：提供**端到端的传输服务**，管理进程间的通信（操作系统层面）。
	- tcp提供可靠传输，有连接和确认，重传。
	- udp是不可靠传输，性能高。
- 网络层：核心是**ip协议**，负责**数据包的路由和转发**，通过ip地址实现跨网络通信，路由器在这一层。
- 网络接口层：最底层，负责在**物理网络**上发送数据帧。将网络层的ip封装为帧，通过**mac地址**在局域网寻址。

# 一.浏览器相关
## 1.Cookie，Seesion和Token是什么？
- Cookie：**存储在浏览器本地**，每次请求自动带上。
	- 不适合存储**敏感信息**。
	- 可以追踪**用户偏好与行为**。
- Session：把用户状态存在服务器，然后给客户端一个`sessionId`。Session是**有状态**的，服务端必须靠它来到Redis里面去**查数据**。
	- 依靠Cookie传递，安全性高但**占用服务器资源**。
- Token：一个**自包含凭证**，包含各种信息，**存储在浏览器**，服务器靠它来识别身份，**不需要**去查库或者Redis。
	- 适合分布式系统，服务端，跨域场景，服务端不用存状态，扩展性好。
	- 实现方式比如JWT。

## 2.JWT
jwt令牌由header，payload，signature构成：
- **header**
	- type：**令牌类型**，即jwt
	- alg：**签名算法**，hs256
- **payload**
	- 存放注册声明，包括jwt主题，接收方，签发时间iat，过期时间exp等
	- 还有**脱敏后的用户信息**。
- **signature**
	- **header+payload+服务端密钥+签名算法**。
最后得到由"."分割的字符串，就是jwt
session保存在redis中
- `user_id`
- `role`
- `exp`

## 3.从输入URL到显示网页发生了什么？
1. 首先，浏览器会**解析 URL**，确定**协议、域名、端口和路径**，比如是 HTTP 还是 HTTPS。
2. 然后进行 **DNS 解析**。浏览器会先查自身缓存、操作系统缓存、hosts 文件，如果没有命中，就请求本地 DNS 服务器。本地 DNS 服务器如果也没有缓存，会继续向根域名服务器、顶级域名服务器、权威 DNS 服务器查询，最终拿到域名对应的 IP 地址。
3. **拿到 IP 后**，浏览器需要和服务器**建立连接**。如果是 HTTP，一般是建立 TCP 连接，也就是三次握手。如果是 HTTPS，还会在 TCP 连接之后进行 TLS 握手，用来协商加密算法、验证证书、生成会话密钥。
4. 在真正发送数据前，数据要经过**网络层和链路层**。主机会根据目标 IP 判断是否在同一个局域网，如果不在，就把数据交给默认网关。这时候会通过 **ARP 协议获取下一跳网关的 MAC 地址**，然后把数据帧发送出去。数据经过多个路由器转发，最终到达目标服务器。
5. 连接建立后，浏览器会**组装 HTTP 请求**，包括请求行、请求头、Cookie 等信息，然后发送给服务器。服务器接收到请求后，可能经过 Nginx、网关、应用服务器，再访问缓存、数据库等资源，最后返回 HTTP 响应，包括状态码、响应头和响应体。
6. 浏览器收到 HTML 后，会开始**解析页面**，**构建 DOM 树**；解析 CSS，构建 CSSOM 树；然后合成渲染树，进行布局和绘制。如果 HTML 中引用了 CSS、JS、图片、字体等**额外资源**，浏览器还会**继续发起请求**加载这些资源。
7. 最后，浏览器执行 JavaScript，完成页面渲染和交互逻辑，用户就看到了完整网页。
