# 一.HTTP
## 1.HTTP各个版本的区别？
### 1.1.HTTP1.0 与 HTTP1.1
- **持久连接**
	- http1.0是短连接，每个请求和响应都涉及三次握手四次挥手，频繁的连接建立与断开带来了高延迟。
	- http1.1是**持久连接**，多个请求可以**复用同一个tcp连接**。
- **host头**
	- http1.0，一个服务器只能搭载一个网站。
	- http1.1，一个服务器可以**搭载多个网站**。

### 1.2.HTTP1.1 与 HTTP2.0
- **多路复用**
	- http1.1虽然是长连接，但是是**多个请求复用同一个tcp连接**，那么如果当前请求没有完成，会**阻塞**其他请求，在请求数量多的情况下很明显。而且http1.1浏览器为了控制资源，同一个域名的tcp连接上限通常是6到8个，这对于多个请求的情况并不友好。
	- http2.0在**同一个tcp连接上可以同时发送多个请求和响应**，**互不干扰**，所以更加高效。
- **二进制帧**
	- http1.1是以**文本形式**传输的，**解析慢且容易出错**。
	- http2.0改为**二进制格式**，机器**解析快**，**数据量和带宽消耗更小**。
- **头部压缩**
	- http1.1对body进行了压缩，但header并没有。
	- http2.0对header采用**hpack**算法进行压缩，**重复字段采用数字代替**，减少了数据的传输量。
- **服务端推送**
	- http2.0加入了服务端推送，服务器再返回结果的时候，可以选择额外发送哪些内容，不过**不好控制**，后面主流浏览器都**弃用**了。

### 1.3.HTTP2.0 和 HTTP3.0
- **传输协议**
	- http2.0基于**tcp协议**。
	- http3.0基于**quic协议**，quic协议是基于**udp**开发的应用层协议。
- **建立连接**
	- http2.0采用**三次握手**来建立连接，如果涉及加密，那么还有一次**tls握手**，整体耗时比较久
	- http3.0是基于quic协议开发的，所以只需要**0个rtt或者1个rtt**。
		- 当客户端**保留了对话前的状态信息**，那么再次发送请求的时候是0个rtt的，也就是可以直接发送加密数据，跳过握手步骤。
		- 当客户端**初次建立连接时**，客户端和服务端**需要一次握手**来完成过程，大大减少了延迟。
- **头部压缩**
	- http2.0使用hpack算法实现头部压缩。
	- http3.0使用**qpack算法**进行头部压缩，**更高效**。
- **队头阻塞**
	- http2.0多路复用即多个请求**复用一个tcp连接**，实际上tcp连接**丢包后**，会**阻塞所有的请求**，直到重新得到包。
	- http3.0是**基于quic协议的**，一个连接可以有**多个数据流**，这些数据流之间**不相互影响**，解决了丢包阻塞的问题。
- **连接迁移**
	- http2.0是基于tcp协议，**tcp连接由（源ip，源端口号，目的ip，目的端口号）四元组组成**，一旦有一项值发生改变，那么这个连接就不能用了。
	- http3.0**支持连接迁移**，只要客户端保留了**64位标识**，只要64位id标识不变，那么即使网络切换也不中断。

### 1.4.QUIC协议
- 基于udp协议
- 在应用层实现了 **流量控制，拥塞控制，重传机制**
- 集成了tls加解密保证安全性
- 一个连接建立**多个数据流**，数据流之间**互不影响**，某个数据流发生丢包，其他数据流不受影响
- 支持**连接迁移**，quic协议使用**64位id（Connection ID）标识**，id不变连接就不中断

## 2.HTTPS
http和https主要有几点不同：
- **端口号**：http是80，https是443
- **路径前缀**也不同 和 **安全性**也不同
	- http建立在tcp之上，是明文传输的
	- https建立在**ssl和tls上**，是安全的
- 由于https更安全，所以也**更受搜索引擎的青睐**
_**HTTPS加密流程：**_
1. 客户端向服务端发送请求，请求建立连接，携带以下参数
	1. 支持的**ssl/tls版本**
	2. 支持的**加密算法**
	3. 一个**随机数**（用于最终密钥的生成）
2. 服务端收到客户端的请求后，发送包含以下参数的请求
	1. **选择**的 ssl/tls版本 与 加密算法
	2. 一个**随机数**（用于最终密钥的生成）
	3. 服务器**证书信息**（包含**公钥**）
3. 客户端收到服务器的请求后，开始验证服务器的证书情况
	1. 比如**证书是否过期**，是否**属于当前域名**，是否**由信任的证书机构签发**
4. 客户端**验证通过后**，开始**生成密钥**，然后**发送给服务器**
	1. 客户端**随机生成一个 密钥**，然后使用证书上的**公钥**进行加密
	2. 服务器**采用自己的私钥**解开密钥（**加密后只有服务器的私钥可以解开**）
5. 最后**生成最终的会话密钥**，使用最开始生成的**两个随机数 + 当前的密钥** 得到最终密钥
6. 最后**建立安全连接**

HTTPS是**混合加密**：
- 握手阶段是非对称加密，安全的生成和交换密钥。
- 数据传输阶段是对称加密。
关于**对称加密和非对称加密**：
- 对称加密：加密和解密是同一把密钥。
- 非对称加密：加密和解密是一对密钥，公钥和私钥。

## 3.HTTP 的 keep-alive 是什么？和 TCP 的 keepalive 是一个东西吗？
- 每次请求都要经历一个http短链接，即请求资源，响应资源，释放连接。资源只能获取一次，所以http的`keep-alive`实现了用**同一个tcp连接来发送和接收多个http请求和应答**，避免了连接建立与释放的开销。通过在**http请求头中的connection：keep-alive来实现**。
- tcp的`keepalive`是tcp层实现的，称为**tcp保活机制**，作用是在tcp连接上检测空闲时连接状态。建立tcp连接后，如果一段时间内没有传输任何数据，那么keepalive会发送数据包来检测连接是否仍然有效。

## 4.HTTP常见状态码
- 1xx，信息类
	- **100 continue**：服务器接收到初步请求，客户端可以继续发送请求体。
	- **101 switch protocals**：服务器同意切换协议，比如从http到web socket。
- 2xx，成功类
	- **200 ok**：请求成功。
	- **201 created**：资源创建成功。
	- 204 no content：没有响应体，比如delete请求的响应结果。
- 3xx，重定向
	- **301 moved permanently**：永久重定向，浏览器更新书签，搜索引擎更新索引。
	- **302 found**：临时重定向，资源暂存在别的地方，下次也访问这个地址。
	- **304 not modified**：没有被更改，配合缓存使用。
- 4xx，客户端错误
	- **400 bad request**：请求错误，比如请求格式有问题。
	- **401 unauthorized**：比如未登录或者token过期。
	- **403 forbidden**：比如用户访问管理员接口。
	- **404 not found**：资源未找到。
- 5xx，服务器错误
	- **500：服务器代码异常**，服务器内部错误。
	- 503 service unavailable：服务暂不可用。
	- 504 gateway timeout：网关超时。

## 5.WebSocket
WebSocket 的工作过程可以分为以下几个步骤：
1. 客户端向服务器发送一个 HTTP 请求，请求头中包含 **Upgrade: websocket** 和 Sec-WebSocket-Key 等字段，表示要求升级协议为 WebSocket；
2. 服务器收到这个请求后，会进行升级协议的操作，如果支持 WebSocket，它将回复一个 **HTTP 101 状态码**，响应头中包含 ，Connection: Upgrade和 Sec-WebSocket-Accept: xxx 等字段、表示成功升级到 WebSocket 协议。
3. 客户端和服务器之间建立了一个 WebSocket 连接，可以进行**双向的数据传输**。数据以**帧（frames）的形式进行传送**，WebSocket 的每条消息可能会被切分成多个数据帧（最小单位）。发送端会将消息切割成多个帧发送给接收端，接收端接收消息帧，并将关联的帧重新组装成完整的消息。
4. 客户端或服务器可以主动发送一个关闭帧，表示要断开连接。另一方收到后，也会回复一个关闭帧，然后双方关闭 TCP 连接。
另外，建立 WebSocket 连接之后，通过**心跳机制来保持 WebSocket 连接的稳定性和活跃性**。
