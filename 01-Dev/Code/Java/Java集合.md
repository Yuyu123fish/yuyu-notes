# 一.单列集合

## 1.ArrayList集合的扩容机制是什么？

初始化阶段调用无参构造，采用**懒加载**策略，底层是一个长度为0的**共享空数组**。第一次调用add方法的时候，会将容量初始化为10，如果10不够就设置为对应的最小容量。后续调用add方法时，会判断当前容量是否满足size+1（也就是最小容量），满足的话就直接赋值，不满足就进行扩容。扩容阶段会将旧容量**右移1位**再加上自身来将新容量设置为原来的1.5倍，如果新容量还比最小容量小，那么会将新容量设置为最小容量。然后会对新容量进行判断，如果比最大容量大，那么会设置为整数最大值。然后会调用Arrays.copyOf来**申请新数组**，**复制**完成后进行赋值。

# 二.双列集合

## 1.HashMap为什么线程不安全？

- 多线程put导致数据覆盖：比如两个线程同时判定可以执行put，且计算出的索引相同时，后写入的线程可能把前一个线程的数据覆盖了。
- jdk1.7采用头插法，形成环形链表：头插法时，多线程同时扩容可能导致链表形成环，后续get导致死循环。（大概就是，a线程在记录完e和next后被挂起，b线程完成更改，此时b已经指向了a，此时a线程恢复，会将a再次头插法，然后将a的next改为b，导致循环）
- put与get并发可见性问题：一个线程put的元素对于另一个线程可能不可见，也就是没有导致可见性。

# 三.并发集合

## 1.ConcurrentHashMap的两个版本的不同？

- jdk1.7：实现原理是分段锁机制，将数据分成多个segment，一般是16个段，每个segment继承`ReentrantLock`，独立加锁，不同segment可以并发访问，提高了并发度。每个segment内部维护一个`HashEntry`**数组**，锁住里面的所有桶，类似于数组+链表。
- jdk1.8：废弃分段锁，实现原理是CAS+synchronized。底层原理和HashMap的一样（数组+链表/红黑树）。锁粒度细化到每个数组元素（桶），使用`synchronized`对链表头节点或者树的根节点加锁，同时使用CAS做无锁操作。并发度更高，引入红黑树增加查询性能。读操作靠`volatile`保证线程间可见性。
- 链表->红黑树

- 链表长度>8且数组元素个数>64
- 红黑树节点<=6，退化为链表

# 四.综合问题

## 1.集合中的fail-fast和fail-safe是什么?

fail-fast和fail-safe是集合遍历遇到并发修改时的两种处理策略。

首先是fail-fast快速失败。快速失败是ArrayList和HashMap等非并发集合的默认机制。其内部维护了一个modCount用来记录修改次数，当进行迭代器遍历时，迭代器自身会维护一个expectedModCount = modCount来在每次迭代时进行校验，如果不相等会立刻抛出ConcurrentModificationException并发修改异常，防止在不可预知的状态下继续操作。

然后是fail-safe安全失败。安全失败是并发集合的策略。比如基于写时复制的集合，在进行写操作时，会拷贝出一个新数组进行修改，写完后替换旧数组。因此，迭代器实际上是在旧数组中进行遍历。这样保证了线程安全，并发修改也不会抛出异常，保证了数据的最终一致性，但不能保证强一致性。
