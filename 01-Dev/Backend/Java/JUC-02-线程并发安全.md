# 一.底层机制

## 1.线程安全的本质

多线程并发访问资源的时候，程序的执行结果是否符合预期，且不会出现数据错乱或逻辑错误，

## 2.synchronized 关键字的底层原理

_**前置知识**_

每个Java对象头部都有一个MarkWord，存储着对象运行时数据，比如哈希码，锁状态信息，GC分代年龄等，锁状态信息标志了synchronized状态。

- `synchronized`关键字是采用互斥方式的对象锁，只有最多一个线程持有该锁，此时其他线程尝试获取就会被阻塞。
- JVM字节码层面：它的底层通过`Monitor`实现，`Monitor`是jvm级别的对象（C++实现），线程获得锁需要使用对象关联`monitor`。synchronized通过 monitorenter 和 monitorexit 两条字节码指令，表示进入临界区，告诉jvm需要加锁和解锁。
- 每个锁对象关联一个`monitor`，内部有三个属性，分别是`Owner, EntryList, WaitSet`

- `Owner`当前持有锁的线程，并且只能关联一个线程。
- `EntryList`等待锁的阻塞线程队列。
- `WatiSet`关联的是处于`Waiting`状态的线程。（可以不说）。

- 具体流程

- 调用 monitorenter 后，jvm会检查 锁对象 的monitor

- 若 owner 为空，线程成为owner
- 若 owner 被占用，线程进入entrylist阻塞

- 当线程执行 monitorexit 时，jvm释放锁，并唤醒entrylist中阻塞的线程，让它们来竞争锁

- 在操作系统层面，要从用户态切换到内核态，向操作系统底层申请互斥量mutex，性能开销很大，因此称为重量级锁。

![](assets/Moniter.png)

### Monitor实现的锁属于重量级锁，你了解过锁升级吗？

Java中`synchronized`有偏向锁，轻量级锁，重量级锁三种形式，分别对应了锁只能被一个线程持有，不同线程交替持有，多线程竞争锁三种情况。

- 偏向锁：一段很长时间内都只被一个线程使用锁，可以使用偏向锁，在第一次获取锁时，会有一个CAS操作，之后线程再获取锁，只需判断markword中是否是自己的线程id即可，而不是开销相对较大的CAS命令。
- 轻量级锁：线程加锁的时间是错开的（也就是没有竞争），可以使用轻量级锁来优化。轻量级锁修改了对象头的锁标志，相对重量级锁性能提升很多。每次修改都是CAS操作，保证原子性。
- 重量级锁：底层使用`Monitor`实现，里面涉及用户态和内核态的切换，进程的上下文切换成本较高，性能较低。

**总结**：锁升级的过程/设计思路 其实是为了适配不同竞争强度下的场景。减小性能开销。

### synchronized锁粒度？

- 实例方法：锁的是当前对象（this）
- 静态方法：锁的是当前类的class对象
- 代码块：锁的是指定对象，粒度更灵活

## 3.谈谈JMM JAVA内存模型

![](assets/JMMJava内存模型.png)

- java内存模型是定义了**共享内存**中**多线程程序读写操作**的行为规范，通过这些规则来规范对内存的读写操作从而保证指令的正确性。目的就是在多线程读写共享数据的时候保证**可见性，有序性，原子性**。
- JMM把内存分为两块，一块是私有线程的工作区域（工作内存），一块是所有线程的共享区域（主内存）。
- 线程跟线程之间是相互隔离的，线程跟线程之间交互需要通过共享内存。

## 4.谈谈你对 CAS 的理解

Compare And Swap，一种乐观锁思想，在无锁情况下保证线程操作共享数据的原子性。执行CAS操作时有三个值，目标内存地址，期待值，新值。

- 执行CAS操作时，会先对内存地址的值进行比较，看看它是不是预期值，如果是就更新为新值，如果不是就会进行自旋操作，即不停重试CAS操作，直到成功为止。
- 因为没有加锁，所以线程不会陷入阻塞，性能较高。
- 如果竞争激烈，自旋频繁， CPU空转影响性能。

_**CAS底层实现**_

CAS底层依赖一个`Unsafe`类来直接调用操作系统底层的CAS指令，它的原子性由操作系统保证，因此比较再交换是一个动作，不会被打断为两个动作。

## 5.谈谈你对 volatile 的理解

一旦一个共享变量（类的成员变量，类的静态成员变量）被volatile修饰后，那么就具备了两层语义：

- 保证线程间可见性。

- 比如一个`while(!flag)`，其中`flag=flase`，那么就会被即时编译器`jit`优化为`while(true)`，不推荐禁用`jit`，因此使用`volatile`之后，会强制保证它是可见的，那么就`jit`就不会对它进行优化。

- 禁止指令重新排序。

- `volatile`修饰的变量会在读，写共享变量时加入不同的内存屏障，阻止其他读写操作越过屏障，从而达到阻止指令重排序的效果。

- 但是不保证原子性，如i++操作仍需同步

![](assets/volatile内存屏障.png)

但是不能阻止写操作从下往上走，也不能阻止读操作从上往下走。使用技巧：

- 写变量让volatile修饰的变量在代码最后位置
- 读变量让volatile修饰的变量在代码最开始位置

## 6.什么是 AQS

![](assets/sync与AQS.png)

- 全称 AbstractQueuedSynchronizer，即抽象队列同步器，它是构建锁或者其他同步组件的基础框架。常见实现类有：ReentrantLock，Semaphore 信号量，CountDownLatch 倒计时锁。
- AQS内部维护了一个FIFO先进先出的双向队列，队列中存储排队的线程。
- AQS内部还有一个属性state，相当于一个资源，默认是0（无锁状态），如果队列中有一个线程成功修改了state为1，则当前线程就相当于获取了资源。
- 在对state进行修改的时候使用了CAS操作，保证多个线程修改情况下的原子性。

实际上，假如在线程释放锁的时候，state被改为1，此时队列中还有等待的线程想拿锁，而又进入一个其他线程来拿到了锁，这就是非公平锁。而对于队列中想要拿锁的线程，这又是公平锁。

**独占模式**：只有一个线程能够访问资源。例`ReentrantLock` 就是独占模式的实现，其中state表示重入次数。

**共享模式**：允许多个线程同时访问资源。例`CountDownLatch` 就是共享模式的实现，其中state表示剩余的计数。

# 二.锁

## 1.ReentrantLock 的实现原理是什么

`ReentrantLock`是一种可重入锁，实现原理如下：

- 同步器：内部有一个`sync`类继承`AQS`，并实现了公平锁`FairSync`和非公平锁`NonfairSync`两个子类。
- 加锁：调用`lock()`时，会委托给`sync`的`lock()`方法。非公平锁直接尝试CAS获取锁，若成功则当前线程独占，失败则进入aqs的重试流程，加入到fifo队列。
- 可重入实现：`reentrantlock`的可重入实现原理是通过修改aqs的`state`字段，比如同一线程获取锁，state++，释放锁就state--，直到state=0，才真正释放锁。同时reentrantlock内部记录了当前持有锁的线程，用于判断当前线程是否重入。

ReentrantLock主要利用CAS+AQS队列来实现。它支持公平锁和非公平锁，比如竞争锁的时候，严格按照顺序获取锁，或者在竞争锁的时候，允许插队来抢占锁资源，两者都继承aqs，主要区别在于`tryacquire`方法：

- 非公平锁：在lock时直接尝试cas获取锁，成功则设置为当前线程，失败则调用`acquire(1)`，在`tryacqeuire()`再次cas重试，不考虑队列。
- 公平锁：在`tryacqure()`中，先检查aqs队列中是否有正在等待的前驱节点，有则返回false，进入等待队列。没有则尝试cas获取锁。

## 2.synchronized 和 Lock 有什么区别

- 语法层面

- `synchronized`是关键字，源码在jvm，由c++实现
- `Lock`是接口，源码由jdk提供，由java实现
- 使用`synchronized`时，退出同步代码块锁**自动释放**，而使用`Lock`时，需要手动调用`unlock`方法进行解锁，不然会导致死锁。

- 功能层面

- 二者均属于悲观锁，都具备基本的互斥，同步，锁重入功能
- `Lock`提供了许多`synchronized`不具备的功能，例如公平锁，可打断，可超时，多条件变量
- `Lock`有适合不同场景的实现，如 ReentrantLock，ReentrantReadWriteLock（读写锁）

- 性能层面

- 在没有竞争时，`synchronized`做了很多优化，如偏向锁，轻量级锁，性能还行
- 在竞争激烈时，`Lock`的实现通常会提供更好的性能

# 三.并发组件

## 1. 聊一下 ConcurrentHashMap

`ConcurrentHashMap`是一种线程安全的高效Map集合

- 底层数据结构

- jdk1.7底层采用 分段的数组+链表 实现
- jdk1.8采用的数据结构跟HashMap1.8一样，数组+链表/红黑树 实现

- 线程安全方式

- jdk1.7采用分段锁机制，将数组分为16个段，每个段都有自己的`ReentrantLock`
- jdk1.8采用`CAS + synchronized`来保证并发安全进行实现

- `CAS`控制数组节点的添加
- `synchronized`只锁定当前链表或者红黑树的首节点，只要hash不冲突，就不会产生并发问题，效率得到提升

## 2. 聊一下 AtomicReference

AtomicReference 是一种基于CAS的并发组件，用于在多线程环境下安全修改对于一个对象的引用，比如对于`AgentStatus`，可以使用`AtomicReference`来安全修改大模型的状态。

## 3. 聊一下 CompletableFuture

java8之前做异步编排的时候，只能使用`Future`，它的缺点是`Future`是基于拉模式的，会阻塞，只能通过`while(true)`来不断轮询，死等。现在的`CompeltableFuture`是基于推模式的（回调）。

- **创建异步任务 (起步)：**

- `runAsync(Runnable)`：没有返回值。
- `supplyAsync(Supplier<U>)`：有返回值。

- **链式处理 (A 跑完跑 B)：**

- `thenApply`：拿到上一步结果，处理后返回新结果（像 Stream 的 map）。
- `thenAccept`：拿到上一步结果，消耗掉，没有返回值。
- `thenRun`：不管上一步结果啥样，只要上一步跑完，我就执行个空任务。

- **异常处理 (兜底)：**

- `exceptionally`：相当于 catch 块，出错了走这里给个默认值。
- `whenComplete` / `handle`：相当于 finally 块，不管成功失败最后都会走。

- **任务编排 (高阶操作，面试必考)：**

- **AND 关系**：`thenCombine` (两个任务都完成，合并结果)，`allOf` (等待数组里所有任务完成)。
- **OR 关系**：`applyToEither` (两个任务谁先跑完用谁)，`anyOf` (数组里只要有一个完成就往下走)。
