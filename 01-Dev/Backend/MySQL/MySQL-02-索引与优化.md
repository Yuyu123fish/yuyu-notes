# 背景知识
1. InnoDB存储引擎每次磁盘I/O读取或写入的大小是一页（16KB）。
2. B+树的结构下每个节点的大小是一页（16KB）。
3. 树的高度 == 每次查询数据时磁盘IO操作的次数。
4. 底层的节点使用双向链表串起来。

## 1.MySQL三层B+树可以存多少数据？
一页是16KB，假设数据类型为bigint，**8字节**，**指针为6字节**，那么一个非叶子节点可以存16×1024/14也就是**一千多个子节点**，而第二层每个节点也可以存位与第三层的叶子节点，而**每行数据假设为1KB**，那么**每个叶子节点（一页）可以存16行**，所以三层B+树能存两千多万条数据。

## 2.MySQL为什么选择B+树作为数据结构
- 树矮，B+树是多叉树，三层B+树能存储两千多万条数据，IO性能好。
- 叶子节点只存key和指针，不存数据。因此内存可以缓存更多索引，命中率高，减少磁盘访问。
- 叶子节点用链表串起来。执行范围查询时找到起点后可以顺着链表查，顺序IO比随机性IO效率高。

# 一.索引
## 1.索引的分类
**按数据结构：**
- B+树索引：多层平衡树结构，**叶子用链表串起来**。既能快速定位单条数据，也能做**范围查询**。InnoDB与MyISAM默认使用。
- 全文索引：**文本分词后建倒排索引**。适合text字段做**关键词搜索**。
- 哈希索引：通过**哈希值存储**，等值查询O(1)。**不支持范围查询和排序**。**Memory引擎**默认使用，InnoDB有**自适应哈希索引**。
- 空间索引：专门处理**地理位置**，支持区域查询与距离计算。

**按存储方式：**
- 聚集索引：叶子节点存**完整的数据**，数据**按照主键顺序排序**。一张表只能有**一个**。
	- 如果使用**无序的UUID**，每次插入都会导致**页分裂**产生大量碎片，严重影响写入性能，而自增主键默认添加在末尾，按顺序写入效率较高。
- 二级索引：二级索引只存储**字段值和主键值**。可能会进行**回表查询**。

**按索引性质：**
- 主键索引：唯一且非空，每张表一个。
- 唯一索引：可以有null值，每张表可以有多个。
- 普通索引：纯粹为了加速。
- 联合索引
- 全文索引
- 空间索引

## 2.索引失效的场景有哪些？
1. 使用了`select *`，mysql可能为了避免回表而不走索引。
	1. mysql可能认为**全表的顺序IO**高于回表时的随机IO。
	2. 最好是使用覆盖索引。
2. 联合索引未遵循最左前缀原则。
	1. 某个字段使用了`<> betwenn %like`之类的**前缀通配符**匹配方式，或者对索引进行了**函数计算**
	2. 由于该字段前面的字段可以正常走索引，但后面的字段可能呈**无序状态**，导致索引无法使用。
	3. 后面的字段如果有**索引下推**(`extra: using index condition`)可以减少回表次数。
3. 出现隐式类型转换，例如字符串和数字的转换。
4. 使用`or`连接条件中，只要一个条件没有索引，则整个查询不走索引。
	1. `index merge`索引合并，同字段走索引，不同字段触发合并，有时优化器评估失败，触发全表扫描
	2. `union`，合并后去重，分别执行**多个子查询**，使用**临时表**来存储和去重，涉及多次内存和磁盘IO，性能开销大，只适合**要严格去重**的场景。
	3. `union all`结果拼接，不去重，性能好。

## 3.最左前缀原则
- **索引下推**：a，b，c三个字段，有a和c没有b时，会字段把c拼到a后面**减少回表次数**。
- 最好是将**选择性高**的字段放在前面。
- Skip Scan：mysql8.0以后，当左侧字段的基数低且查询语句只用了右侧的字段，mysql会**自动拼上左侧的字段**，使用索引进行查询。
- 查询优化器：即使查询语句没有按照索引的字段**顺序**进行查询，mysql也会自动调整为对应的字段顺序。

## 4.回表查询的代价
- 因为通过**二级索引拿到的id**可能在**聚集索引里面不连续**，所以会产生**大量随机IO**，性能急剧下降。而**顺序IO通过磁盘预读**，性能较高。
- **回表次数太多**，MySQL优化器还可能进行**全表扫描**，因为是顺序IO，性能可能更高。

## 5.MySQL中索引一定有效吗？如何排查
- 优化器成本计算：是否走索引取决于优化器对走索引和全表扫描的**IO成本和CPU成本**的计算。
- 使用`Explain`进行计算。
- 索引失效场景（上文）。

## 6.索引过多的代价
- 写性能下降：每次写入数据都涉及索引的更新，性能负担增大。
- 优化器负担大：优化器每次需要对多个所索引的成本进行判断，生成执行计划的时间变成。
- DDL变慢：DDL涉及树的重建，耗费时间变久。
- 索引占用空间大：表里面数据很多的时候，如果主键是随机且很长的uuid，那么占用的存储空间很大，对磁盘和内存的开销大，影响`BufferPool`的存储容量。

# 二，优化
## 1.如何定位慢查询？如何优化？
1. 定位慢查询：
	1. 开启慢查询日志，设置`long_query_time`为2秒来定位。
	2. 通过`explain`生成执行计划，观察是否使用索引，预估的行数和`extra`信息。
2. 优化：
	1. 检查是否命中索引，`explain`中`key`为`null`说明没有走索引，需要添加和优化
	2. 避免全表扫描，为`where,order,join`建立索引
	3. 避免`select *`，函数操作和隐式类型转换
	4. 数据量过大就进行分库分表

## 2.Explain执行计划
- `type`：连接类型，从好到差：system，const，eq_ref，ref，range，index，all。至少达到range级别，避免all全表扫描
	- `system`，const：表中匹配了一条数据->主键，唯一索引
	- `eq_ref`：常见于join，驱动表的每一行对于被驱动表只有一条数据
	- `ref`：用到了索引，可能是非唯一索引或者走了最左前缀，可能匹配了多行
	- `range`：索引范围扫描，从一个节点开始沿着链表扫
	- `index`：全索引扫描
	- `all`：全表扫描
- `key`：使用的索引，为null就是没用
- `key_len`：索引键长度，越短越好，可以判断是否使用了联合索引的全部或者部分
- `rows`：预估需要扫描的行数，越短越好
- `extra`：额外信息。
	- `using index`，覆盖索引
	- `using where`，使用where过滤
	- `using index condition`，索引下推
	- `using where`和`using index condition`的区别是前者是拿到结果后再使用`where`进行过滤，后者是在回表取结果前使用了索引下推减少了回表次数。

## 3.深度分页问题
`select * from user order by id limit 1000000, 10;`问题是 MySQL 要扫描并丢弃前面大量数据。
	1. 用覆盖索引**先查 id**，再回表查询完整数据，也叫**延迟关联**。
1. 使用游标分页，比如 where id > last_id limit 10（缺点是只能连续翻页）。
2. 业务上**限制最大页数**，避免无限深翻页。或者**无限滚动**。
3. 搜索类场景可以考虑 ES，但不是普通业务 SQL 的首选方案。
