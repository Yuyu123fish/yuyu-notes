# 背景知识
1. InnoDB存储引擎每次磁盘I/O读取或写入的大小是一页（16KB）。
2. B+树的结构下每个节点的大小是一页（16KB）。
3. 树的高度 == 每次查询数据时磁盘IO操作的次数。

# 一.索引
## 1.索引失效的场景有哪些？
1. 使用了`select *`，mysql可能为了避免回表而不走索引。
	1. mysql可能认为**全表的顺序IO**高于回表时的随机IO。
	2. 最好是使用覆盖索引。
2. 联合索引未遵循最左前缀原则。
	1. 某个字段使用了`<> betwenn %like`之类的匹配方式，或者对索引进行了函数计算
	2. 由于该字段前面的字段可以正常走索引，但后面的字段可能呈**无序状态**，导致索引无法使用。
	3. 后面的字段如果有**索引下推**(`extra: using index condition`)可以减少回表次数。
3. 出现隐式类型转换，例如字符串和数字的转换。
4. 使用`or`连接条件中，只要一个条件没有索引，则整个查询不走索引。
	1. `index merge`索引合并，同字段走索引，不同字段触发合并，有时优化器评估失败，触发全表扫描
	2. `union`，合并后去重，分别执行**多个子查询**，使用**临时表**来存储和去重，涉及多次内存和磁盘IO，性能开销大，只适合**要严格去重**的场景。
	3. `union all`结果拼接，不去重，性能好。
