# 一.事务
## 1.事务的四大特性
ACID四种特性：**原子性+隔离性+持久性=一致性**
- A，原子性：一个事务中的所有操作，要么同时成功，要么同时失败。由`undolog`实现。
- C，一致性：事务执行前后的数据状态是一致的。由 原子性 + 隔离性 + 持久性 共同实现。
- I，隔离性：事务之间不可以互相影响，数据库有支持并发修改的能力。由 `MVCC`和锁 实现。
- D，持久性：事务结束后，对数据库的影响是永久的。由`redolog`实现。
 
## 2.并发事务特性
 - 脏读：事务读到了别的事务**还未提交的数据**。
 - 不可重复读：事务**两次查询同一条数据**，得到的**结果不一样**。
 - 幻读：事务两次执行同一条范围查询，得到的行数不一致，也就是有别的事务插入或删除了。
	 - 可以通过MVCC+临键锁来解决。

## 3.事务隔离级别
- 读未提交：事务看到的是没有提交的数据，有脏读的问题。
- 读已提交：只能看到别的事务提交了的数据，但是有不可重复读的问题。每次读取都**生成一个新**的`readview`。
	- 大厂为什么选择 **不可重复读** 作为事务隔离级别：
		- 可重复读的`undolog`形成版本链，**空间占用大**
		- 可重复读涉及锁竞争，涉及**锁等待和锁竞争**
- 可重复读：事务两次查询同一条数据，得到的结果是相同的，通过`undolog`和`mvcc`实现，**初次读取**的时候生成一个`readview`，后来的事务都在上面进行操作。**很大程度**上避免了幻读，但没有完全避免。
	- 快照读：通过`mvcc`实现。
	- 当前读：使用记录锁+间隙锁实现。
- 串行化：事务串行执行，没有并发性。

# 二.三大日志
## 1.binlog归档日志
`Server`层的日志，主要用于 **数据备份** 和 **主从同步**，或者`Canal`伪装MySQL从节点订阅`binlog`用于**保证数据一致性**。
binlog是一个二进制文件，记录了**表结构和表修改**的日志，不会记录查询操作。
binlog有三种格式：
- statement：记录每条数据修改的sql，**主从复制根据sql重现**，遇到比如`now`的**动态函数**，会出现数据不一致的情况。
- row：记录**真实的行变更**，缺点是文件**体积大**。
- mixed：**混合使用**，遇到动态函数换成`row`。

## 2.redolog重做日志
`InnoDB`引擎层的日志，实现了事务的**持久性**，主要用于**故障恢复**。
_**背景知识**_
mysql启动的时候会给bufferpool申请一片**连续的内存空间**，包含多个16KB的页，比如数据页，索引页，自适应哈希索引等等，当一条数据被更新的时候，内存中的这一页被标记为**脏页**，并以`redolog`的形式记录下来（WAL，先写日志），并在合适的时机刷到磁盘中。
脏页刷到磁盘是随机IO，而**写到redolog是顺序io**，顺序IO性能比随机IO好。
_**redolog的作用**_
redolog是**物理日志**，记录了**某个数据页做了什么**。由两个文件构成，采用**循环写**，新的修改不断覆盖旧的修改，当旧的修改没有被刷到磁盘时，会**阻塞整个mysql**然后将旧的修改刷到磁盘。
断电恢复的时候由于存了redolog，记录了对应的**物理变化**，所以直接读取redolog即可。

## 3.undolog回滚日志
`InnoDB`层的日志，实现了事务的原子性，主要用于**事务回滚**和**MVCC**。
事务发生回滚时直接读取`undolog`，**保证了事务的原子性**。
一次更新操作产生的undolog，就会带有一个**roll_pointer指针**和一个**trx_id事务id**。通过**roll_pointer串成版本链**，通过trx_id可以得到该记录被哪个事务修改的。

# 三.MVCC
## 1.ReadView在MVCC如何实现？
**事务的四个关键字段：**
- m_ids：事务创建`readview`时，**活跃（启动了但没提交的事务）的事务id列表**。
- min_trx_id：m_ids的**最小值**。
- max_trx_id：创建`readview`时应该传递给**下一个事务的id值**，也就是全局事务中的最大事务id+1。
- creator_trx_id：**创建**该`readview`的事务id。
**其中还有两个隐藏列：**
- trx_id（**修改记录的事务id**）：当一个事务对某条记录**修改**的时候，会将该事务id放到trx_id。
- roll_pointer：每次对该**记录发生修改**时，会把旧纪录写到`undolog`，然后**指针指向该记录形成版本链**。

**流程如下：**
- 如果trx_id小于min_trx_id，那么说明这个版本的记录是在这个`readview`之前的事务生成的，所以是**可见的**。
- 如果trx_id大于等于max_trx_id，那么说明这个版本的记录是在当前`readview`之后的事务生成的，所以是**不可见的**。
- 如果当前trx_id在min_trx_id和max_trx_id之间，则需要判断其是否在m_ids，即活跃事务id列表中：
	- 如果trx_id在m_ids之中，说明当前`readview`创建的时候，该事务还没提交，不可见。
	- 如果trx_id不在m_ids之中，说明当前`readview`创建的时候，该事务已经提交，可见。

## 2.MVCC如何实现读已提交和可重复读？
对于读已提交和可重复读，他们都是基于`undolog`和`readview`实现的：
- 读已提交：此隔离级别每个`select`都会生成一个新的`readview`，也意味着事务期间多次读取同一条数据，可能出现数据不一致的情况，因为其他事物可能修改了该记录。
- 可重复读：此隔离级别只在启动事务的时候生成一个`readview`，后面的修改全在此`readview`基础上进行，保证了事务期间读取到的都是事务启动前的数据。

# 四.锁
## 1.MySQL中的锁的分类
### 1.1.按锁粒度区分
- 行锁：InnoDB 的行锁是基于索引实现的。如果 where 条件没有命中索引，就会扫描大量记录并加锁，导致锁范围扩大，效果上类似锁表。命中了索引就锁对应id的那一行或者周围。
	- 记录锁：只锁这一行记录。
	- 间隙锁：锁住记录中间，防止插入。
	- 临键锁：锁记录和前面的空隙。
- 表锁：如果没有命中索引，就锁住整张表来进行扫描。
- 全局锁：常用于全局逻辑备份，防止备份期间出现数据不一致的情况。

### 1.2.按模式分
- 共享锁：允许多个事务同时读。
- 排它锁：独占，不让其他事务碰。

## 2.MySQL发生死锁如何解决？
- 自动处理：
	- 开启死锁检测，自动回滚，通常是`undolog`最少。
	- 锁超时等待，超时自动回滚。
- 手动干预：
	- 找到阻塞的线程id并杀死。
