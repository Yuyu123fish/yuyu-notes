# 二.三大日志
## 1.binlog归档日志
`Server`层的日志，主要用于 **数据备份** 和 **主从同步**。
binlog是一个二进制文件，记录了**表结构和表修改**的日志，不会记录查询操作。
binlog有三种格式：
- statement：记录每条数据修改的sql，主从复制根据sql重现，遇到比如`now`的动态函数，会出现数据不一致的情况。
- row：记录真实的行变更，缺点是文件体积大。
- mixed：混合使用，遇到动态函数换成`row`。

## 2.redolog重做日志
`InnoDB`引擎层的日志，实现了事务的**持久性**，主要用于**故障恢复**。
_**背景知识**_
mysql启动的时候会给bufferpool申请一片**连续的内存空间**，包含多个16KB的页，比如数据页，索引页，自适应哈希索引等等，当一条数据被更新的时候，内存中的这一页被标记为**脏页**，并以undolog的形式记录下来（WAL，先写日志），并在合适的时机刷到磁盘中。
脏页刷到磁盘是随机IO，而**写到redolog是顺序io**，顺序IO性能比随机IO好。
_**redolog的作用**_
redolog是物理日志，记录了某个数据页做了什么。由两个文件构成，采用**循环写**，修的修改不断覆盖旧的修改，当旧的修改没有被刷到磁盘时，会**阻塞整个mysql**然后将旧的修改刷到磁盘。
断电恢复的时候由于存了redolog，记录了对应的**物理变化**，所以直接读取redolog即可。

## 3.undolog回滚日志
`InnoDB`层的日志，实现了事务的原子性，主要用于**事务回滚**和**MVCC**。
事务发生回滚时直接读取`undolog`，保证了事务的原子性。
一次更新操作产生的undolog，就会带有一个**roll_pointer指针**和一个**trx_id事务id**。通过**roll_pointer串成版本链**，通过trx_id可以得到该记录被哪个事务修改的。
