# 一.RabbitMQ基础
## 1.Rabbit模型
核心模型：Producer -> Exchange -> Queue -> Consumer
- 生产者不直接把消息发给队列，而是先发给交换机，交换机再根据路由规则把消息投递到不同队列。

常见交换机：
- direct：精确匹配 routingKey。
- fanout：广播给所有绑定队列。
- topic：通配符匹配 routingKey。
- headers：根据消息头匹配，较少使用。

RabbitMQ优点：
1. 路由能力强，交换机模型灵活。
2. 支持 ACK、持久化、生产者确认、死信队列、延迟队列等机制。
3. 适合业务系统异步解耦。
4. 社区成熟，使用成本相对较低。
缺点：
5. 吞吐量一般不如 Kafka、RocketMQ。
6. 大规模消息堆积时性能压力较大。
7. Erlang 技术栈，对 Java 后端同学来说排查底层问题成本略高。

适合场景：订单创建后异步发短信、发邮件、扣库存、通知其他系统。

## 2.RabbitMQ如何保证消息不丢失？
以下环节均会发生消息丢失的情况：
- 生产者将消息没有到交换机。
- 交换机的消息没有到消息队列。
- 消息队列的消息丢失。
- 消费者没有收到消息。
生产者 -> 交换机 -> 消息队列 -> 消费者

- **生产者确认机制**：
	- 消息发送到消息队列后，消息队列会返回一个`ack`到生产者表示消息成功接收。
- **持久化消息**：
	- **RabbitMQ默认内存存储消息**，如果未开启持久化可能导致消息丢失。
	- 可以开启 交换机持久化，队列持久化，消息持久化。
- **消费者确认**：
	- 消费者收到消息并成功处理后，需要返回`ack`给消息队列表示已成功处理。消息队列收到`ack`后才删除消息。
- **失败重试**：
	- 可以利用`Spring`的失败重试机制，或者是手动处理失败消息。阶梯重试后，达到一定的失败次数（例如3次或4次）后就放入异常交换机交给人工处理。

## 3.RabbitMQ的消息重复消费如何处理？
**消费者突然宕机**，或者消费者**处理完消息后突然网络异常**，都可能导致消息重复消费，解决方法如下：
- **使用乐观锁机制**，`UPDATE order SET status = '已支付' WHERE id = 123 AND status = '待支付'`，保证只更新一次。
- 每条消息设置**唯一的消息id**（使用Redis的Set集合，设置过期时间，消费前检查是否已经存在）。
- **数据库唯一索引**，如果重复消费则存在插入同一条数据的情况，则数据库插入失败，不做重试直接`ack`即可。
- **去重表**：维护一张去重表，在处理前先根据消息的唯一id查询是否处理过。

## 4.RabbitMQ死信队列
延迟队列：进入队列的消息会被**延迟消费**，场景是：**超时订单，限时优惠，定时发布**。
延迟队列 = ttl + 死信交换机
当一个队列中的消息满足以下条件之一就可以被称为死信：
- 消费者显示拒绝或者返回不确认**声明消费失败**，并且消息的`requeue`设置为`false`。
- 消息是一个**过期消息，无人消费**。
	- 消息所在队列有`ttl`。
	- 消息本身有`ttl`。
- 要投递的队列消息**堆积满了**，**最早的消息可能成为了死信**。

## 5.RabbitMQ如何实现延迟消息？
通过 **死信队列 + 消息ttl** 间接实现：
1. 创建一个普通队列，不设置消费者，为**队列中的消息设置过期时间ttl**。
2. 将消息发送到这个队列而**不消费**。
3. 不消费的消息过期变成死信，死信消息会被**路由**到绑定的**死信交换机**中。
4. **真正的消费者**从死信队列中消费消息，实现延迟效果。

## 6.消息堆积怎么解决？
当 消费者 消费消息的速度 < 生产者 生产消息的速度 时，消息会**不断堆积在队列**中，直到达到上限，此时**新的消息会成为死信，可能会被丢弃**，这就是消息堆积问题。
解决消息堆积主要有三种思路：
- 增加**更多消费者**，提高消费速度。
- 在消费者开启**线程池加快消息消费速度**。
- 扩大队列容积，提高堆积上限，采用**惰性队列**。
	- 声明队列为lazy
	- 基于磁盘存储，存储上限高，但受限于**磁盘IO**，时效性低。
**惰性队列**
- 接收到消息后**直接存入磁盘**而非内存。
- 消费者要消费消息时才会**从磁盘中读取并加载到内存**。
- 支持数百万条消息存储。

## 7.RabbitMQ 高可用机制了解过吗？
生产环境下，使用集群来保证高可用性，比如 普通集群，镜像模式，仲裁队列。
**普通集群**
- 会在各个节点间共享部分数据，包括交换机，队列元信息。不包含队列中的消息。
- 访问集群中某个节点时，如果队列不在该节点，会传递到队列对应的节点并返回。
- 队列所在节点宕机，队列中的消息就会丢失。
**镜像模式**（本质是主从模式）
- 交换机，队列，队列中的消息会在各个mq的镜像节点之间同步备份。
- 创建队列的节点称为**该队列的主节点**，备份到的其他节点叫做**该队列的镜像节点**。
- 一个队列的主节点可能是另一个队列的镜像节点。
- 所有操作都是主节点完成，然后同步给镜像节点。
- 主节点宕机后，镜像节点会称为新的主节点。
**仲裁队列**
- 与镜像队列一样都是主从模式，支持主从同步。
- 使用非常简单，没有复杂配置。
- 主从同步基于Raft协议，强一致。
