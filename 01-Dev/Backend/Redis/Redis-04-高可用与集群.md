# 一.主从集群
## 1.主从集群实现原理
1. **建立连接与全量同步**：从节点连上主节点后，主节点会返回服务器id，传输对应的RDB文件。这期间新增的命令会放到缓冲区（replication buffer，每个节点一个），RDB传输完成后再传输增量命令完成全量同步。
2. **命令传播**：全量同步完成后，主从会建立长连接，期间主节点的写命令会发给从节点。期间还会互相发送心跳包确认存活。
3. **断线重连与增量同步**：主节点内部有一个环形缓冲区（覆盖写），通过服务器id和offset来判断是增量同步还是全量同步。

## 2.为什么有多节点架构？
单节点的缺点：
- 单redis节点有**内存上限**，无法支持数据量大的场景，内存大小成为瓶颈。
- 单节点有**宕机风险**，可用性低。
- 大量客户端与单节点通信会让redis的**带宽成为瓶颈**。

主从的优势：
- 支持**读写分离**，主节点负责写，然后同步给从节点，从节点负责读，分摊主节点的压力。
- **数据一致性**：如果不是主从，全部节点都允许读写操作的话，不好保证一致性，实现很复杂。

## 3.主从架构的缺点？
- 读写分离带来的**延迟**问题。
- 如果从节点过多，主节点全量同步的压力过大，可以变为 **主-从-从** 架构。
- master节点宕机需要有**故障恢复**机制。
- 只有一个master节点，无法应对**高并发写**的请求。

# 二.哨兵机制
## 1.哨兵集群的作用
- **监控**：不断检查主节点的从节点的健康状态。
- **自动故障恢复**：主节点宕机会自动选举新的主节点，也会去尝试让宕机的主节点恢复。
- **通知**：故障发生时会通知所有节点。

## 2.哨兵集群如何监控主从集群？
哨兵集群基于**心跳机制**，每秒向每个实例发送ping命令
- **主观下线**：某个哨兵发现某个实例下线，则认为它主观下线。
- **客观下线**：超过一定数量的哨兵发现某个实例主管下线，则会认为它客观下线。

## 3.master宕机后如何选举新的master？
1. 判断slave与master的断开时间长短。
2. 判断slave的优先级。
3. 判断slave的offset值。
4. 判断slave的运行id大小。

## 4.哨兵机制的问题
哨兵机制只解决了故障恢复的问题的，但对于单个节点的数据压力过大的问题并没有解决。
- 无数据分片，单个节点存储巨量数据。
- 无法应对高并发写。

# 三.Cluster分片集群
## 1.分片集群实现原理
主要原理是**负载均衡与快速查找**。
- **数据分片机制（哈希槽）**：集群固定有16384个槽位，每个节点管理固定的槽位，使用`CRC16(key)`对16384取模来找到节点。
- **节点通讯与架构（去中心化）**：节点间通过`gossip`协议不断发送消息来确认存活状态，交换节点信息。
- **客户端寻址**：客户端发送的信息都会发送到正确的节点。
- **自带容灾**：分片集群自身实现了高可用，不需要哨兵机制。
	- **主从复制**：每个节点可以挂载多个从节点，实现读写分离。
	- **故障发现**：通过`gossip`协议确认存活状态。半数以上确认宕机后确认为客观下线。
	- **自动选举**：主节点宕机后会自动选举新的主节点。
	- **水平扩展**：可以新增主节点实现水平扩展。

# 四.高可用
## 1.BigKey问题
BigKey是**String类型的value超过1MB**；**Hash，List，Set，ZSet类型元素超过5000个**的Key。

BigKey的危害：
- 极度占用内存且导致内存分布不均匀。
- 网络IO占用大，可能会增加延迟甚至导致阻塞。
- 客户端超时，处理大Key时间长，可能导致客户端超时。

可以扫描Redis的key来发现BigKey。

如何解决BigKey：
- **直接删除**很危险，会**阻塞主线程**（Redis单线程执行）。
- 可以使用`unlink`，将它从主线程的键空间摘掉，**后台异步线程会释放内存**。
- 或者**大Key拆成小Key**，比如1000个元素的List拆分为五个200元素的List。

## 2.HotKey问题
HotKey，热点Key是被**高并发访问的Key**，如果很多请求全部打到Redis，由于Redis是**单线程**处理命令，可能导致Redis无法处理其他命令导致**阻塞**，把**CPU吃满**，集群模式下导致**流量倾斜**。

如何发现HotKey？
- **业务预判**：零成本，提前预判，适合可预期的运营活动。
- `--hotkeys`扫描：是扫描全部Key，速度比较慢，而且只能看某个时间点。
- `monitor`：实时监控，但是会消耗50%的性能，适合**测试**。
- **客户端埋点**：比较**靠谱**，就是需要改造客户端，**性能损耗低**。

如何解决HotKey？
- 多级缓存：Redis里面加一层本地缓存，比如`caffeine`，先读本地缓存，减轻redis压力。
- 热点Key拆分：按用户id取模，去不同节点查数据。
- 读写分离
- 限流降级
