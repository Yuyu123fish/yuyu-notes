# 一.缓存相关问题
## 1.缓存三兄弟
- 缓存穿透：大量的请求取查询数据库里面**不存在的数据**，**无法命中缓存**，给数据库带来巨大压力。
	- **入口检验**：参数过滤，参数合法性放在最前面，小于0直接拒绝，不合法直接返回。
	- **布隆过滤**：请求先到布隆过滤器里面检验一遍，但存在**误判**可能。
		- 基于布隆过滤器特点，即对key多次计算哈希值在分布到位图中，存在误判可能，对于01的位图，还存在无法删除key的情况，改成存对应位置出现次数的好点。
	- **缓存空对象**：数据库查了没有，就往缓存里面写一个空标识，设置**较短的过期时间兜底**，防止同一个不存在的key反复穿透。
- 缓存击穿：一个被高并发访问的key突然失效，大量请求打到数据库，给数据库带来巨大压力。
	- **互斥锁重建缓存（保证一致性）**：只让第一个查询的线程拿到互斥锁，然后完成缓存的重建，对于没拿到锁的就阻塞等待得到结果。
	- **逻辑过期返回旧值（保证可用性）**：不设置ttl，而是在字段里面设置过期时间，获取时检测。
		- 方案一：当前线程查询到为旧值后，获取到分布式锁，然后独自完成缓存重建并返回。其他没拿到锁的线程就返回旧值。
		- 方案二：后台单独起一个线程，定期检查，快过期了就去重建缓存，其他查询的线程直接返回就行。
		- lua保证在删除锁动作（判断+删除）的原子性。
- 缓存雪崩：大量带有ttl的key同时过期，导致查询无法命中缓存，请求打到数据库，给数据库带来巨大压力。
	- **随机化ttl**，给ttl设置随机值，防止大量key同时过期。
	- **服务降级与熔断**：发现数据库压力过大就采用降级策略，返回**友好的提示信息**，并进行处理。
	- **本地缓存兜底**：对于读多写少的热点数据，使用`caffeine`和`guava`加一层本地缓存。

## 2.Redis如何保证缓存与数据库的一致性？
### 2.1.不可用方案
- 本质上都是**两个线程交替更改**带来的不可用，**一个线程的两个操作被另一个线程插入了**。
- **先更新缓存，再更新数据库**：更新缓存可能涉及复杂计算，不如直接删除高效。
	- a把缓存更新了，此时b完成更新缓存和数据库，a再更新数据库，导致数据不一致。
- **先更新数据库，再更新缓存**：
	- a把数据库更新了，此时b完成更新数据库和缓存，a再更新缓存，导致数据不一致。
- **先删缓存，再更新数据库**：
	- a先把缓存删除了，b发现没缓存去查数据库，此时还没写入缓存，然后a完成更新数据库，b又写入缓存导致数据不一致。

### 2.2.可用方案
- **CacheAside旁路缓存**：**先更新数据库，再删除缓存**。实时一致性高的情况下首选。
	- 极端情况：**缓存刚好过期**，a发现没缓存去查数据库，此时b又更新数据库删除缓存，a又写入缓存导致数据不一致。
- **延迟双删**：先删缓存再更新数据库，延迟一小段时间后再删除缓存。延迟消息可以使用消息队列或者定时任务。
- **Binlog异步更新**：Canal伪装mysql从节点，订阅主节点的binlog，binlog发生修改后通知消息队列，消费者负责处理。消息必须按顺序消费，且需要保证幂等性。

### 2.3.强一致性方案
业务不能容忍任何不一致，比如金融场景，可以加**分布式读写锁，读读不互斥，读写互斥**。但是性能下降，且写操作会阻塞所有操作，高并发不合适。

## 3.缓存删除失败怎么办？
- **重试机制**：将删除失败的key加入消息队列，异步补偿重试直到成功。
- **设置缓存过期时间**：即使删除失败也会过期，可能导致短时间不一致。
- **binlog监听**：使用canal伪装mysql从节点，当binlog发生变化，canal便发送消息到消息队列，让线程删除。
- **延迟双删**：更新前删一次，更新后删一次，降低不一致窗口。

## 4.强一致性方案
业务不能容忍任何不一致，比如金融场景，可以加**分布式读写锁，读读不互斥，读写互斥**。但是性能下降，且写操作会阻塞所有操作，高并发不合适。

# 二.分布式锁
## 1.Redis如何实现分布式锁
Redis实现分布式锁的核心是：
- 加锁用`set key value EX seconds NX`，-EX设置过期时间，防止死锁；-NX保证互斥。
- 解锁用**lua脚本**先校验再删除，保证**原子性**。
	- 因为要先判断锁是不是自己的，是自己的才删除，如果不使用lua脚本，那么**判断和删除**是两个操作，没有原子性，那么这之间自己的锁可能被**其他由于超时释放锁的线程**拿到锁然后释放了，造成了并发问题，可能造成数据不一致或者资源冲突。

加锁：
- 客户端执行`set lock_key uuid EX 30 NX`，Redis检查key是否存在，不存在则成功设置key，存在则返回nil，加锁失败。
解锁：
- 客户端执行lua脚本，`get lock_key`获取当前值，判断值是否等于自己的uuid，相等则del lock_key，解锁成功，不相等则返回0，说明锁不是自己的。

## 2.Redis的分布式锁有哪些问题？
业务逻辑层：
- 锁过期但业务没跑完。
- 误删别人的锁。
架构高可用层：
- 主从同步延迟。
	- 主节点还没把锁数据同步到从节点就挂了，造成从节点没有锁信息。
- 单点故障问题。
	- 单个Redis节点有个致命问题，节点挂了分布式锁就没了。

## 3.RedLock红锁是什么？
用**多个独立的redis实例**，客户端在**大多数实例上加锁成功**才算拿到锁。用来解决redis主从架构下锁丢失的问题。
加锁：
- 客户端记录当前时间戳为`t1`，然后向所有实例发送带`ttl`的加锁请求。
- 统计成功数量和时间戳`t2`，发现成功数量大于向上取整的一半且`t2-t1`小于`ttl`，则加锁成功。否则向所有节点发送解锁请求，即`del`命令。
