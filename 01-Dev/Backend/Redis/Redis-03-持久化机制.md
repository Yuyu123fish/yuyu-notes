# 三种持久化机制
## 1.RDB持久化
**快照持久化**，把某个时间的redis内存数据`dump`成一个**二进制文件**，特点是**文件体积小**，适合做**备份容灾恢复**。缺点也很明显，两次RDB持久化之间的数据如果redis挂了，那么**数据就丢了**。

`save`：同步的RDB持久化操作，会**阻塞主线程**。
`bgsave`：主进程`fork`出一个子进程用来持久化，采用`copy-on-write`写时复制操作。
- `fork`操作本身是阻塞的，会带来额外的开销，且数据越大开销越多。
- `copy-on-write`写时复制，主进程执行读操作，会访问共享内存。主进程或子进程修改该页时，内存会复制该页。大数据集情况下多次复制会对性能造成影响。

## 2.AOF持久化
**日志持久化**，每条命令都**追加到文件里**，数据安全性高，最多丢失一秒数据，缺点是**文件体积大**，恢复的时候要把所有命令重新执行一遍，数据量大的时候恢复很慢。

AOF流程：
1. 命令追加：所有的命令全部写到**AOF缓冲区**。
2. 文件写入：调用`write`函数将AOF缓冲区的内容全部写入到AOF文件就返回，此时数据还在**内核缓冲区**，还未持久化到磁盘。
3. 文件同步：根据AOF**同步策略**来将内核缓冲区的数据持久化到磁盘。
**同步策略**：
- `appendfsync always`，主线程调用`write`后立即同步持久化到磁盘。数据安全性好，但一直阻塞性能最差。
- `appendfsync everysec`，后台线程每秒同步一次，性能好，最多丢失一秒数据。
- `appendfsync no`，由操作系统决定什么时候进行持久化，性能最好，但数据丢失不可控。
AOF重写：
- Redis再后台写一个新的AOF文件，但**体积更小**，AOF重写会执行大量的**写入操作**，为了避免对主进程造成影响，会使用**子进程进行**。
- AOF重写时增量命令会写入到**AOF重写缓冲区**，重写完成后**追加到末尾**。

AOF为什么在执行命令后记录日志：
- 不用检查**命令合法性**。
- 执行完命令后再记录，不会**阻塞**当前正在执行的命令。

## 3.混合持久化（4.0新值）
aof重写的时候直接转为RDB文件，然后增量命令以AOF的形式追加到末尾。恢复时先加载RDB，再回放AOF命令，增强了恢复速度和安全性。

流程：
- **写处理阶段**：客户端执行写命令，Redis立即更新内存并将命令写到AOF缓存区。
- **持久化触发阶段**：AOF文件大小达到阈值，触发AOF重写（bgrewriteaof）。
- **文件构建阶段**：**子进程**将当前**内存数据以RDB形式写入到AOF文件开头**。**父进程**继续完成读写命令，**增量数据写到重写缓冲区**。重写完成后，将重写缓冲区的增量命令**追加**到aof文件末尾。
- **数据恢复阶段**：先加载RDB部分快速恢复，再回放AOF增量命令。

缺点：
aof文件里的**rdb数据是压缩格式**，不再是aof格式，可读性较差。需要消耗额外的cpu进行rdb压缩和解压。
