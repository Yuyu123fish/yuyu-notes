# 一.Workflow与三种范式
## 1.Workflow与Agent
`Workflow`是一种确定的流程图，每一步的**逻辑是硬编码**的，执行的路线是明确的。
- 好处是**行为完全可以预测**，容易测试，出了问题方便排查。
- 坏处是**灵活度低**，如果出现没有预测到的地方容易进入**死胡同**。

`Agent`则相反，它把下一步做什么的决策权交给了大模型，告诉它目标，它会自己判断是否该使用工具，得到结果后是否继续。
- 坏处是行为不确定，同样的输入路径可能得到不同的结果，线上**出了错误也很难排查**。

能用`workflow`解决的问题就不要用`agent`，因为`agent`的行为是概率性的，同样的输入可能走不同的路径。

## 2.三种范式
- ReAct：Reasoning + Acting，最常见的一种，推理和行动交替进行。
- Plan-and-Execute
	- 先用一个LLM（规划器）做规划，然后由另一个LLM（执行器）负责执行。
	- 如果最开始的方向是偏离的，可能造成无法挽回的结果。
	- 同时也具有动态调整的能力，比如在进行某一步之前，会额外进行另一次规划外的操作。
- Reflection：反思
	- 每完成一步之后就进行一次反思，判断做的好不好，是否符合预期。
	- 升级版是`Reflexion`，它会把反思总结和失败的原因，改进建议作为额外的上下文传给下一次尝试。
三者结合效果最好，但代价是成本和时间增加。

# 二.Agent能力
## 1.什么是Agent Harness Engineer？
Agent Harness Engineer是**做Agent运行底座的人**。

Harness 可以理解成“**把大模型包起来、让它稳定做事的一整套执行框架**”。也就是不只是调 LLM API，而是负责：
- Agent 的**执行循环**，比如 ReAct、Workflow、Graph Runtime、State Machine；
- **工具调用系统**，比如 Tool Calling、MCP、权限控制；
- **上下文管理**，比如 memory、context 压缩、长期/短期记忆；
- **模型路由**，比如不同任务走不同模型；
- Guardrail，比如安全策略、工具权限、终止条件；
- Trace / Eval / 日志观测；
- Agent SDK，让其他业务工程师能方便接入 Agent。

Agent Harness Engineer 更偏 **AI Infra + Backend**，不是单纯写 prompt，也不是单纯调模型，而是建设 Agent 的运行框架。它负责把 LLM、工具、记忆、状态机、上下文管理、权限、评估和观测整合起来，**让 Agent 能稳定、可控、可观测地在生产环境里执行任务**。

## 2.Trace是什么？
Trace是**一次 Agent 执行过程的完整链路记录**。
把 Agent 每一步都记下来，比如：
- 用户输入是什么；  
- 拼给模型的上下文是什么；  
- LLM 调用了几次；  
- 每次模型输出了什么；  
- 调用了哪个工具；  
- 工具参数是什么；  
- 工具返回了什么；  
- 哪一步耗时最长；  
- token 消耗多少；  
- 有没有重试、报错、死循环；  
- 最终答案是怎么来的。
它的作用是**排查和优化 Agent**。
