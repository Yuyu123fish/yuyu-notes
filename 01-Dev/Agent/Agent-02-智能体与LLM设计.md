# 一.上下文处理
## 1.上下文爆炸了(过长)怎么处理？
**先保存状态，检测到工具结果/上下文过长后，不要硬塞进 LLM，而是做替换、摘要、让 Agent 换策略**。你还提到了日志场景下可以**分块摘要**，这个也很关键。

上下文爆炸本质上是 **Agent 的历史对话、工具结果、检索结果、执行状态都不断进入 prompt**，导致超过模型窗口，或者虽然没超过但噪声太多、效果变差。

我的处理思路是**分层控制**：
- 第一，进入模型前做 **token 预算管理**，不能所有内容都无脑塞进去。
- 第二，对历史对话做**短期记忆压缩**，只保留**关键目标、约束、已完成步骤和重要结论**。
- 第三，工具调用结果如果太长，不直接返回全文，而是先做**裁剪、分页、过滤、摘要**，或者只返回**结构化的关键信息**。
- 第四，如果是日志、文档这类大结果，可以分块处理，先检索/过滤，再使用模型做摘要，最后只把**和当前问题相关的片段**给模型。
- 第五，**保留执行状态**，比如当前**任务阶段、已调用工具、失败原因**，这样即使压缩上下文，也不会丢掉 Agent 的任务进度。

# 二.Tool，MCP与Skill
## 1.ToolCalling工具调用
`ToolCalling`发生在模型与工具之间，是单词调用格式的规范。
- 我们提供给模型工具的名称，描述，参数列表，工具调用的结果。
- 模型会在特定时机触发工具调用拿到结果。
- 工具是针对当前的模型或者框架写的，不够标准化。

## 2.MCP模型上下文协议
对工具的标准化封装，在模型之间是可以**通用**的。
- 客户端连上服务端后**自动发现工具列表**，从而进行调用。
- 缺点就是出了问题不好排查，要去查MCP服务器。

## 3.Skill
Skill是一套**执行模板**，包含**执行指令、脚本和模板**。比如 代码审查 skill，里面涉及多个步骤和调用多个MCP工具。
比如 数据分析skill，可能会涉及到先从数据库拉数据，再用Python做趋势分析，再按模板写分析报告。这个过程中涉及多个MCP的调用，每个MCP具体调用里面又涉及多个工具的调用。

## 4.如何做一个Skill？
skill是给Agent使用的一份任务操作手册。
它负责告诉Agent在遇到某一类任务时：
- **按照什么流程做**？
	- 适应场景和输入输出。
	- 执行步骤。
- **注意什么边界**？
	- 约束条件。
- **失败了怎么办**？
	- 失败处理和示例。

# 三.高可用
## 1.Agent服务突然挂掉了怎么办？
Agent服务挂掉了，可以从三层考虑：**请求层降级熔断，状态恢复，服务高可用**。
- **降级熔断**：
	- 内部问题：
		- 比如当前 Agent run 调模型、查知识库、调工具时，如果依赖**超时或异常**，要有 timeout、retry、fallback。不能让请求一直卡住。
		- 对于用户侧，可以返回“**当前任务失败/正在恢复/请重试**”，而不是一直无响应。
	- 外部问题：
		- 如果是某个**外部工具挂了**，可以**熔断这个工具**，暂时不再调用，避免拖垮整个 Agent。
- **状态恢复（更关键）**：
	- 执行过程中的上下文，工具调用结果，当前的步骤和任务状态，都可以记录为当前的总体状态，记为一个**checkpoint**，存储在外部。Agent服务挂掉以后可以快速恢复。
- **高可用部署**：
	- 对于**多节点部署**，网关/Nginx 要有健康检查，发现某个 Agent 服务挂了，就把**流量摘掉**，**转发到其他节点**。
	- 对于**异步任务**，比如文档解析、长任务执行，可以用 Redis Stream / MQ，任务只有 **ACK** 后才算完成，服务挂了以后其他 worker 可以继续消费 pending 任务。

## 2.智能体陷入死循环的原因是什么？如何解决？
本质上是因为ReAct是一个“思考->行动->观察->再思考“的循环结构，如果**没有明确的终止条件**，或者**每一轮的观察结果不能推动状态的变化**，就可能一直反复执行。
- **工具调用失败**：超时，报错，返回空结果，智能体不断重试同一个工具。
	- 可以显示告诉模型工具调用失败，让模型意识到可以换策略等等。
- **任务目标不清晰**：智能体不知道什么时候算完成。
	- 提示词里明确获取足够信息就可以总结答案。
	- 或者使用额外的判断节点，让模型思考是否可以进行总结。
- **缺少最大调用次数**。
	- 兜底：节点进入次数。
	- 第二层防护：模型调用次数。
	- 第三层防护：`Service`里面预留接口，当前允许用户自己中断。
- **上下文没有及时更新**，比如工具调用后结果没有注入上下文，智能体又反复执行这一步。
	- 每次调用工具后，都要把工具名，参数，结果注入上下文，同时使用状态机约束状态的转移流程。
- **记忆或召回不准确**，注入了干扰信息，导致模型**误判**任务没有完成。
	- 对于召回的文本做**相似度阈值和TopK限制**，去重和排序，低相关内容不注入。
	- 记忆写入阶段要做**白名单和黑名单过滤**，避免把系统说明，临时问题，错误信息写入长期记忆。
	- 必要时也可以把召回的内容**标注来源**，让模型知道哪些是记忆，哪些是知识库，哪些是工具调用的结果。

## 3.原子状态机解决了什么问题？
- 使用`AtomicReference<>`，使用它的`getAndSet(AgentStatus next)`，类似CAS的思想，会先判断当前状态是否是期望状态，然后再进行更改，防止并发修改下的冲突问题。
- 同时`transactionTo()`包裹`getAndSet()`方法，适合加入日志查看状态变化。
- 同时`getAndSet()`会返回旧值，适合日志调试。

## 4.会话锁怎么做？
- 使用令牌桶算法做SSE连接槽限流，同一个用户同时最多跑两个对话。
- 同一个对话发起多个请求时，只有一个请求可以拿到`Redis`里面的会话锁，指令`SET NX EX 120`。
- 而Redis锁失效时，比如**Redis挂掉**，会由上面的**原子状态机做兜底策略**，第二个请求进入Agent但是此时状态是`RUNNING`，就也可以正常拦截到。
